## 二、项目核心事实 (不要动摇, 不要再查证)

### 硬件平台
- **PSG1 v0.2**: 单通道方波 + 4-bit 音量 (TLC7524 衰减器)
- **时钟 CLK = 64 kHz** (v0.1 验证的甜点, 覆盖 C3-C8)。不是 125kHz。
- **控制: FT232H (CJMCU 板)**, 12 根线 (v0.3 加 D7=A1):
  - C0-C7 (ACBUS, MPSSE 0x82 命令) = 数据总线 (写 period / 方波控制字 / 噪音控制字)
  - D4 (ADBUS bit4, 0x80) = LE (period 写选通, HC373)
  - D5 (ADBUS bit5, 0x80) = A0 (方波 HC374 CP, 写音量+占空比)
  - D6 (ADBUS bit6, 0x80) = RST (复位, 低有效)
  - D7 (ADBUS bit7, 0x80) = A1 (噪音 HC374 CP, 写噪音控制字)  ← v0.3 新增
- **MPSSE 注意**: D1(TDI)/D2(TDO) 被强制拉高, 不能写低。D0-D7 里 D0/D3/D4/D5/D6/D7 可写, D1/D2 不可。

### 信号链
```
FT232H C口(period) → U1(LS373 电平转换 3.3V→5V) → U8(HC373 period锁存) → U2/U3(HC161 计数器)
FT232H C口(音量)   → U1(LS373) → U10(HC374 音量锁存) → U6(TLC7524 DB4-7)
X1(有源晶振) → CLK → U2/U3/U5
U5(HCT74 toggle) → U6.TLC7524 Pin15/REF(方波输入) → R-2R衰减 → Pin1/OUT1(输出)
                 → [耦合电容 → 运放 → 喇叭]
```

**输出链路铁律**: TLC7524 OUT1 之后**必过耦合电容 + 运放**才到喇叭。
net / 原理图里的 U7(喇叭) 只是**示意图占位**, 真实输出链路是
OUT1 → 电容 → 运放 → 喇叭。不要再问"有没有运放""能不能推喇叭"——必有, 是设计前提。

**TLC7524 接法铁律 (踩过的大坑, 不要重犯)**:
- **正向接法 (当前, 正确)**: REF=方波输入 / OUT1=输出 / RFB=GND / OUT2=GND
- **反接电压模式 (已弃用, 错误)**: OUT1=输入 / REF=输出 / RFB 悬空
  → 实测噪音叠加方波 + 音量失效。datasheet SLAS061D 电压模式只保证 OUT1≤2.5V 线性,
    且寄生电容随数字码跳变 (30↔120pF) 致振铃噪音。
- 用户熟悉 R-2R (Gigatron 经验), 接法以用户实测为准, 不要拿 datasheet 的运放闭环方案质疑。

**总线"串扰"真相 (2026-07-02 修正, 不要再误判)**:
- 之前以为 LE/A0 总线复用会串扰 (写 period 时音量被改), 在脚本里加了"互关隔离"逻辑 (写 period 前关 A0, 写音量前关 LE)。**这是误诊**。
- 真正根因: **用户接线错误** — RST 线错接到了 LE 位置, 复位脉冲乱触发锁存/计数器, 产生各种奇怪行为。
- 修正后 (Main_2026-07-02.net): LE/A0/RST 各走独立 LS373 通道, 设计本身没问题。
- 教训: **遇到奇怪行为先查硬件接线, 不要急着改软件加防御逻辑。** 脚本里的隔离逻辑多余但无害, 可保留。

### TLC7524 引脚 (实测 DIP-16, 已据 net 文件核对)
```
Pin1=OUT1  Pin16=RFB(悬空,电压模式)
Pin2=OUT2  Pin15=REF(→喇叭)
Pin3=GND   Pin14=VDD(+5V)
Pin4=DB7   Pin13=WR(→GND,常写入)
Pin5=DB6   Pin12=CS(→GND,常选通)
Pin6=DB5   Pin11=DB0(→GND)
Pin7=DB4   Pin10=DB1(→GND)
Pin8=DB3   Pin9=DB2(→GND)
```
电压模式: VO = VI × (D/256), D=vol<<4 (vol 0-15 → D 0-240)。

### 音量接线: DB4-DB7 高 4 位 (已论证最优, 不要再改)
- **必须连续 4 位**: 权值 1:2:4:8 才能让 16 档等距铺开。非连续位 (如 DB7,5,3,1)
  权值跳跃, 16 值挤成 4 团, 实际只剩 4 档有效, 反而浪费位数。
- **必须接高段**: DB4-7 满量程 D=240 (94% 幅度, 4.69V)。
- 4 位硬件 DAC 物理上限即线性 16 档; 要 SN/AY 那种对数曲线需软件查表 + DAC 接满 8 位,
  属另一套架构, 留待 v0.3+。

### 物料约束 (重要, 选芯片前查)
- **芯片随便用**: 库存表 (CLAUDE.md 7.4 节) 里的 74/CD/TLC 芯片都可用, 包括模拟开关 CD4066、
  锁相环等。**没有"纯数字不用 XX"的限制** — 不要编造这种约束。
- **电阻电容紧缺**: 用户手头缺少各种阻容元件。需要外接 RC 的方案 (如 CD4046 PLL 的 VCO 要 R1/R2/C1)
  要明确告知"需要外接电阻电容", 让用户判断是否方便。
- **数字 vs 模拟开关的选用**: 
  - 切数字信号 (进 DB 口) → 数字多路器 HC153/HC157 优先 (保真数字逻辑)
  - 切模拟/高频信号 (进 REF 口, 如 HC161 的 Q 抽头做乘法调制) → 模拟开关 CD4066 更合适 (保真谐波)
  - 选用依据是信号保真需求, 不是"纯数字原则"

## 三、硬件踩坑 (从 v0.1 development-log 带过来, 不要重犯)

1. **电平转换必需**: FT232H 3.3V < 74HC VIH(3.5V), 所有 FT232H 信号必须经 LS373 转换。
   **A0 也要过 LS373** (v0.1 的 gate 就是没转换才不响)。
2. **toggle 用 HCT74 不是 HC74**: HC74 高频翻转失效 (v0.1 实测 A6+ 消失)。
3. **period 数据走 C 口**: D1/D2 是 MPSSE TDI/TDO, 写 0 被强制拉高。
4. **每片芯片就近加 0.1μF 去耦电容**, 所有 GND 共地。

详见 `psg_voice/PSG1 v0.1/docs/development-log.md` 第 10 章 "硬件调试实录" (7 个 bug 全记录)。

## 四点五、项目顶层定位与长期愿景

**核心方向**: 用 74 系列门电路 + AI 辅助设计流程 (glm + iverilog), 从零设计数字音源芯片。
不是 1:1 复刻专用 IC, 而是实用主义原型——每片芯片都经仿真+WAV 试听确认有用。

### 音源方向 (当前重点, 从声音最直观的入手)

| 阶段 | 内容 | 状态 |
|------|------|------|
| PSG1 v0.1-v0.2 | 方波 + 4bit 音量 + ADSR | ✅ 已上板出声 |
| PSG2 v0.3 | 方波占空比 + LFSR 白噪音 | CH0 上板验证通过 (16 片) |

可以直接"翻译"成 74 门电路或查表机程序, 不需要自己发明音源算法。

### 查表机谱系 (⭐ SCC 的前置技术, 2026-07-03 认知修正)

**关键洞察**: SCC/FM 不是"更复杂的音源电路", 是**专用的音频查表计算机**。
相位累加 (HC161 只能计数, 不会自己调度) + 多通道分步读 SRAM (读 step→累加→查波形→乘音量)
需要状态机指挥, 而**状态机 = 微程序 ROM 驱动 = 查表机/CPU 的核心架构**。

**SCC 绕不开查表机**。HC161 轮询 ROM 出控制指令是唯一可行的状态机实现 (HC138 译码太死板,
组合逻辑爆炸)。项目里已有三机谱系, SCC 是 fsm_player 和 39040 的中间形态:

| 项目 | 片数 | 架构 | 能力 | SCC 适用度 |
|------|------|------|------|-----------|
| **fsm_player** | 9 | 顺序读 ROM, 无分支 | PCM 回放/LED 动画/固定序列 | PCM ✅ (地址自主递增), SCC ❌ (无循环/无多地址) |
| **SCC 专用查表机** | ~14-16 (规划) | 微程序 ROM + 音频数据通路 | 相位累加+查表+乘音量, 指令集固定 | ⭐ 目标 (够用不臃肿) |
| **39040cpu** | 22 | 通用图灵机 (取指/译码/分支/ALU) | 任何计算 | ✅ 通用 (SCC/FM 当程序写) |

**三机的本质区别**:
- fsm_player: 地址线性递增, ROM 内容直出 (最简回放)
- 39040: 地址可跳转 (分支), ROM 当指令译码执行 (通用计算)
- SCC 专用机: 介于两者 — 要循环 (相位累加) + 多地址访问, 但指令集固定 (不需通用译码)

**ym2413.c 的 NCO (`phase += step; idx = phase>>10; carrier = conv_vol[level][idx]`)
本质就是查表机的一个循环**。硬件化 = 微程序 ROM 驱动这几步反复执行。

**修正后的开发顺序** (原"39040 暂缓"认知已过时):
```
PSG v0.3 ✅ (纯硬件音源, 已完成)
   │
   ▼ 纯组合/时序逻辑到头了, 要状态机才能继续
fsm_player 上板验证 (9 片, PCM 回放) ← 最简查表机
   │
   ▼ 加分支/循环 → SCC 专用查表机 (或直接用 39040)
SCC = 查表机 + DAC + 音频微程序
   │
   ▼
FM (查表机 + 二级调制程序)
```

### CPU/时序方向 (查表机成熟后的自然延伸)

- **39040cpu**: 74 系列 CPU 项目 (RTL + ROM + 接线表已有, 22 片图灵完备零隐藏门)。
  **不是"暂缓的副业", 是 SCC/FM 的前置技术** — 通用查表机通了, SCC/FM 就是写程序。
- **fsm_player**: 9 片最简查表机 (顺序读 ROM), PCM 回放专用。39040 的"前辈"。
- **prog_timer**: 4 片定时器, v0.1 时期的 iverilog 工具链验证项目 (development-log 第 14 条)。

### 方法论 (所有子项目通用)

全实例化 (不是 yosys 综合) → AI 辅助 RTL → iverilog 仿真 → WAV/数据验证 → 接线表 → 上板。
PSG 验证过的流程, SCC/CPU/FM 都能套用, 只是芯片数和复杂度递增。

  - **噪音通道 rev.a 突破** (2026-07-07, HC374 + 模拟噪声注入):
    - 原版 HC164 LFSR 上板锁死 → 改 LS374 (原装) 一样锁死 → **任意移位寄存器都会锁死 (固有特性)**
    - 用户创新: HC00 门3 + 2×0.1μF 电容构成张弛振荡器 (实为模拟噪声源), D[0] 注入持续噪声
      → LFSR 退化成"模拟噪声采样器", **永不锁死** (D[0] 真随机, 不可能收敛)
    - 实测稳定几分钟无问题, 频繁切频率挡不锁死 (原 RC 上电脉冲方案会锁死)
    - RTL: `psg_noise_374_v03.v`, 接线表: `docs/wiring-table-rev-a.md`
    - **这绕过了业界标准防锁死补丁 (RC 灌种/全0检测/TR-909启动), 用模拟特性根治数字死穴**
  - **噪音方波同源认知** (2026-07-07): 方波和噪音在 74 离散逻辑里是同源的, 都是"计数器+D触发器",
    区别只在 D 接哪个信号 (tc_hi=方波, pe_n=采样竞争=噪音). 改一根线就能切换.
  - 新增 RTL 原语: `rtl/cd4070.v` `rtl/hc164.v` `rtl/hc153.v` (均据 datasheet 查证引脚)

## 五点五、AI 辅助 74HC 芯片设计流程 (已验证跑通, 2026-07-02)

> v0.3 全程用此流程, 从架构规划到上板搭线, 遇到的问题都是接线/门类型/FT232H 电平,
> **没有架构硬伤**。证明 AI 辅助数字芯片设计流程完整可行。

### ⚠️ 核心原则: 全实例化, 不是综合 (与 yosys 路线本质对立)

业界有另一条路线 (Pepijn de Vos / PCBFlow): 用 yosys/ghdl 把行为级 Verilog
**自动综合**成 74 门。但他们的 breathing LED demo 就要 5 片, 门利用率极低, 无实用性。

**本项目走相反路线**:
- **全实例化, 无隐藏门**: RTL 直接实例化 hc161/hc164/cd4070 等真实芯片模型,
  不写抽象行为级让综合器猜。每一片芯片、每一个门都由人决定。
- **74 TTL 高利用率**: 每片芯片的每个门都有用途 (如 HC00 剩余门借给噪音通道, 不浪费)。
- **iverilog 只做仿真, 不做综合**: 验证功能对不对, 不替你决定用几片芯片。
- **人脑做架构, AI 做验证**: 选什么芯片/怎么连线由人定, AI 负责 RTL 建模+仿真+WAV 试听+接线表。

这就是为什么本项目 7 片能做出 v0.2 (带音量方波), 16 片做出 v0.3 (占空比+噪音),
而综合路线做同等功能可能要几十上百片。

### 流程环节 (每一步都不可省)

1. **架构规划** (design.md)
   - 讨论通道/波形/芯片选型, 记录决策和弃用方案
   - 查 datasheet 确认芯片能力 (引脚/功能/约束), 不凭记忆
   - 库存核对 (查库存表, 不臆断)

2. **RTL 行为级建模** (psg_xxx.v)
   - 实例化真实芯片模型 (hc161/hc164/cd4070...), 不写抽象黑盒
   - 行为级表达组合/时序逻辑 (TLC7524 用 assign 等效)
   - 注释标清每片芯片用途 + 寄存器位分配

3. **仿真验证** (tb/psg_xxx_tb.v)
   - iverilog + vvp (见第六章命令)
   - 打印关键信号 (LFSR 状态/计数器/翻转次数), 算占空比/频率
   - 用 VCD 波形辅助调试

4. **WAV 试听** (tb/psg_xxx_wav_tb.v)
   - 把 RTL 输出按 64kHz 采样写成 WAV (16-bit PCM, mono)
   - 试听确认音色/频率是否符合预期 (耳朵是最终验收标准)
   - **重要**: 仿真"功能正确"≠ 听感可用 (周期噪音仿真过但试听死锁)

5. **接线表** (wiring-table.md)
   - 每片芯片逐脚标信号来源/去向, 据 datasheet 引脚号
   - 寄存器位分配要看驱动 py (别臆测, 见沟通纪律第 2 条)

6. **上板搭线** (用户操作)
   - 按接线表插面包板, AI 不直接参与
   - 出问题先查: 接线错误 > 门类型搞错 > FT232H 电平 > 设计问题

### 关键教训 (流程中踩的坑)

- **datasheet 必查 PDF 原文**: HC164 MR 无置位、HC153 上下半部共享地址、CD4070≠HC86 引脚、
  HC374 D/Q 交错排列 — 凭记忆全错, 搜索总结也不可信 (自相矛盾), **必须读 PDF 原文**
- **同型号接线优先于 datasheet**: 项目里已上板验证的同芯片接线 (如 HC374 U6) 应作为首选参考,
  datasheet 是补充验证. (但要注意旧接线表也可能抄错, 如 U6 D/Q 标反 — 实测最权威)
- **WAV 试听必做**: 周期噪音仿真"正确"但试听死锁, 耳朵才能发现
- **寄存器位看驱动**: 音量在 bit0-3 还是 bit4-7, grep 驱动 py, 别猜
- **绑定方向要算清**: HC161.CET 门控会让噪音频率和音高成反比 (反直觉)
- **max-length taps 要查表**: Q7⊕Q5 周期只有 30, 要 Q7⊕Q5⊕Q4⊕Q3 才周期 255

### ⚠️ 仿真的根本盲区 + 面包板的发现价值 (2026-07-02 实证, 2026-07-07 深化)

iverilog 是**逻辑仿真器**, 不是电路仿真器。有些音色现象从原理上就无法模拟:

- **mode 白噪** (pe_n 进 7474): 根源是 **7474 采样竞争** (HC00 延迟让 pe_n 跳变沿偏移到 clk 上升沿附近,
  7474 采样落在建立时间窗内, 采到新值还是旧值不确定). ⚠️ 噪声源是采样竞争, 不是 pe_n 信号本身
  (TC/pe_n 直接接喇叭只听到刺耳高频变形方波, 不是噪声 — 必须经过 7474 D 触发器采样才出噪声).
- **Q 抽头音色** (Q0/Q1): 根源是 HC161 输出的**非理想边沿瞬态** + **reload 混叠**.
  Q0=小号 (高泛音), Q1=萨克斯 (低泛音). 音色来自 Q 本身的模拟特性, **不是 TLC7524 乘法调制**
  (TLC7524 只衰减音量, 直接把音响接 Q 芰点就是这个音色).
  泛音会随编程 period 循环变化 (reload 边沿与 Q 跳变沿的相位关系循环, 类似车轮效应).
- iverilog 的 `assign Q=q_reg` 是理想跳变, 看不到模拟特性 → 这些音色仿真复现不了

**流程的第三条腿 — 面包板插针探索**:
1. iverilog 仿真: 验证你想得到的逻辑
2. WAV 试听: 验证仿真结果的音色
3. **面包板插针: 发现你想不到的现象** ← 意外音色的来源

v0.3 的 Q 抽头音色、mode 白噪、LFSR 模拟注入都是面包板意外发现的, 不是设计出来的。
PSG/SID 经典芯片的 glitch 音色很多也是工程师硬件上"乱接"发现的。
**面包板不是"验证仿真", 是"发现仿真发现不了的东西", 两者不可替代。**

### ⭐ 实践创新原则 (2026-07-07 用户确立, 最高优先级)

**有些问题还是得用户想办法创新, 不要纠结, 在实践中找答案。**

反面案例 (AI 纠结门级逻辑, 用户一个实操解决):
- LFSR 锁死问题: AI 反复纠结 "HC86 做 OR 要几个门"、"XOR 换 XNOR"、"CD4078 全 0 检测",
  全是纸上推演, 没一个解决实际问题. 用户加两个 0.1μF 电容到 HC00 反相器, RC 变成模拟噪声源,
  **根治**.
- 关键转折: AI 归因"国产 HC374 质量差", 用户纠正"原装 LS374 一样锁死" →
  **任意移位寄存器都会锁死, 是固有特性, 不是芯片问题**. 业界标准做法 (RC 灌种/全0检测/TR-909启动)
  都是补丁, 没根治. 本项目创新: D[0] 注入模拟噪声, LFSR 退化成采样器, 不可能锁死.

教训:
- 仿真/理论给方向, **实测给答案**
- 纠结门级逻辑时, 跳出来想"有没有完全不同的路"
- 74 芯片的模拟特性 (延迟、瞬态、振荡) 是宝藏, 不是干扰
- **实践中的创新 > 书本上的标准做法**
- AI 卡壳时应提醒用户"要不要换个思路", 而不是继续打补丁

## 七、芯片查证纪律 (重要, 别凭记忆)

### 7.1 不确定芯片特性就查 datasheet, 不要猜
- 凭记忆写芯片模型/方案 → 引脚错、功能错, 浪费一整轮仿真。
- 查证渠道: WebSearch + datasheet PDF (TI/Nexperia 官方优先)。

### 7.2 已查证的关键事实 (别再质疑/重查)
- **HC164 MR 脚**: 异步清零, 强制 Q0-Q7 全 LOW, **无置位/置数能力**。与 CP/DSA/DSB 无关。
  → MR 接 rst_n 后, RST 释放 = 全 0 = 白噪死锁 (Q7⊕Q5=0 永远 0)。这是真硬件行为。
- **HC153**: 双 4 选 1, **上下半部共享地址脚 A/B** (TI datasheet 明示)。
  → 不能上半部选频率、下半部选绑定。绑定改用 HC161 的 CET/CEP 门控。
- **CD4070 vs 74HC86**: 都是四异或门, 但**引脚排列不同** (4000 系列门2/4 的 A/B 顺序反)。
  → 不可复用 hc86 模型, 必须独立建 cd4070.v。
- **CD4021**: 异步并行预置 (P/S=HIGH 立即载入), **但只引出 Q6/Q7/Q8**, 缺 Q0/Q5。
  → 做 LFSR 时 taps 受限 (周期反馈要 Q0, 它没有)。

### 7.3 LFSR 死锁问题 (v0.3 已解决, RST 灌种法)
- HC164 无预置能力, RST 后全 0 → 白噪死锁; 周期模式 Q0 自反馈也会死锁/退化。
- **解法 (已实现 + 仿真验证)**: RST 期间 serial_in=1 持续灌种子,
  8 拍后 LFSR=0xFF, 白噪 max-length 自持永不归零。借 HC00 剩余门反相 RST, 0 额外芯片。
- 试过 CD4078 全0检测: 能防白噪死锁但周期模式仍循环退化; 最终砍周期噪音, 不需 CD4078。
- **max-length LFSR 性质**: 种子非 0 时白噪永不归零 (Q7⊕Q5⊕Q4⊕Q3 周期255, 仿真验证)。
- **绑定模式坑 (已修)**: 旧方案 HC161.CET=square_tc 会让噪音频率和音高成反比 (音越高噪音越低)。
  修正: bind=1 时 noise_clk 直通 square_tc, 噪音频率=2×方波频率, 音越高越密。
  - **方波通道白噪模式 (bug 变 feature, 2026-07-02 硬件实测)**: 把 7474 FF1 的 D 输入
    从 tc_hi 改接 pe_n (HC00 反相 tc_hi 输出), 方波通道意外输出高频白噪。
    原理: HC00 真实传播延迟使 pe_n 跳变沿偏移, 7474 用 clk 采样时落点看似随机 → 白噪。
    这是**硬件物理延迟产物, iverilog 仿真复现不了** (理想 HC00 零延迟)。
    实现: +1 片 HC86 二选一 (mode=0 方波 / mode=1 白噪)。
    局限: 音色批次相关不可控, 不如 LFSR; 但 0 额外噪音芯片, 占空比链仍生效。
    详见 design.md 5.8 节。
  (各位不同步跳变)。方波无此问题 (取 TC→toggle 是边沿干净信号, DAC 当衰减器)。
  但锯齿/三角的 Q 直喂 TLC7524.DB 会输出毛刺尖刺。
  TLC7524 自带的 WR 锁存是**电平透明型** (透明期间毛刺直通), 挡不住。
  **必须用 HC273 边沿寄存器**在 clk 沿采样稳定值 (Gigatron 同款解法, 多 1 片)。
  详见 design.md 2.1 节 + 5.5 节 PSG3 预览。
