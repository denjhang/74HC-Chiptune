# PSG3 v0.4 设计文档

> **YM2413 风格地址/数据复用总线 + 多通道寄存器统一寻址**
> 创建时间：2026-07-07
> 状态：🚧 **总线协议 + 两通道 + 驱动全部上板验证，VGM 预录制 + 噪音乐器完成**，待扩通道/锯齿三角

## 一句话目标

PSG2 v0.3 的"每寄存器一根选通线 + 一片 HC374"模式到极限了（FT232H 12 根线用满，再扩通道选通线爆炸）。PSG3 v0.4 **仿 YM2413**，把地址和数据复用到 8 位总线，A0 区分地址/数据，地址空间从 4 个寄存器扩到 **8 个（可扩到 256）**，为 PSG3 多通道（3 方波 + 噪音 + 锯齿/三角）和未来 SCC 铺路。

**核心原则**：FT232H 物理接线**几乎不动**（仍 12 根线），只改控制线的**语义**。地址一对一选通 374，无译码器（最简、最"74 门原教旨"）。

---

## 一、为什么改总线（PSG2 v0.3 的瓶颈）

### PSG2 v0.3 现状（每寄存器独占选通线）

| FT232H | 信号 | 选通对象 |
|--------|------|---------|
| C0-C7 | D0-D7 数据 | 直连 HC373(period) + HC374(方波控) + HC374(噪音控) 的 D 端 |
| D4 | LE | HC373 period 锁存使能 |
| D5 | A0 | 方波 HC374_CP |
| D6 | RST | 复位 |
| D7 | A1 | 噪音 HC374_CP |

**问题**：
1. 每加一个寄存器 = 占一根 FT232H 选通线 + 一片 HC374
2. FT232H 可靠 GPIO 只有 C0-C7(8) + D4-D7(4) = 12 根，PSG2 v0.3 已用满
3. 扩到 3 方波 + 噪音 + 锯齿/三角 = 8+ 寄存器，选通线根本不够
4. SCC 5 通道 = 15+ 寄存器，彻底爆炸（design.md 8.6.5 节已预判"寄存器爆炸"）

### YM2413 风格复用（PSG3 v0.4 方案）

**两拍写一个寄存器**（地址 + 数据），8 位总线复用：
- 地址空间：8 位 = 256 个寄存器（PSG3 只用 8 个，余量巨大）
- FT232H 线数：**12 根，不变**（语义改变）

| FT232H | PSG3 v0.4 信号 | vs PSG2 v0.3 |
|--------|---------------|-------------|
| C0-C7 | **D0-D7 复用总线**（地址+数据） | 数据线不变（仍 8 根） |
| D4 | **A0**（0=地址, 1=数据） | LE → A0（语义变） |
| D5 | **/WR**（写脉冲，上升沿锁存） | A0 → /WR（语义变） |
| D6 | /RST | 不变 |
| D7 | **/CS**（片选，事务边界，**必须 FT232H 控制**） | A1 → /CS（语义变） |

> ✅ **物理接线零增减**（仍 12 根线），只是 D4/D5/D7 的控制语义重新定义。
> ⚠️ **/CS 不能接 GND**——它是事务保护壳，负责消除"短暂地址值"中间态（见 2.2 节）。必须由 FT232H 的 D7 控制：事务期间 /CS=0，事务外 /CS=1。

## 二、总线协议（仿 YM2413，两拍写）

### 2.1 地址编码 = 独热码（⚠️ 关键，非二进制）

"不用地址解码，8bit 直接控制 8 个 374"意味着地址是**独热码（one-hot）**，不是二进制数：

| 寄存器 | 独热码地址（总线上放的值） | 选中的地址位 |
|--------|--------------------------|------------|
| reg0 | 0x01 = 00000001 | ADDR[0] |
| reg1 | 0x02 = 00000010 | ADDR[1] |
| reg2 | 0x04 = 00000100 | ADDR[2] |
| ... | ... | ... |
| reg7 | 0x80 = 10000000 | ADDR[7] |

每位地址一对一选通一个 HC374，无译码器。上位机写寄存器时，地址字段放 `(1 << reg_n)`。

> **vs HC138 译码**：HC138 把 3 位二进制地址译成 8 选 1（可扩展，但要译码器芯片）。独热码用 8 位地址直接选通，0 译码器，最简。代价是地址空间不能压缩（8 个寄存器占满 8 位），但 PSG3 8 个寄存器刚好够，SCC 阶段（15+ 寄存器）再上 HC138 译码。

### 2.2 写时序（两拍写 + /CS 事务包，已仿真验证 ✅）

**雅马哈协议精髓**：一次完整写入 = 两次总线操作（写地址 + 写数据），用 /CS 把整个事务包起来。**两次写入构成一次无歧义的完整事务**——寄存器只跳变一次（直接变成最终值），没有任何中间态。

```
写寄存器 reg[n] = data（完整事务）:

/CS = 0                    ← 事务开始 (选中芯片)

第 1 次写 (写地址):
  bus  = (1 << n)          独热码地址上总线
  A0   = 0                 声明: 这是地址
  /WR  ↓ 再 ↑              写脉冲 (上升沿锁地址)
  → 地址锁存 HC374 在 /WR↑ 锁住独热码地址 (A0=0, data_strobe 无效, 不锁数据)

第 2 次写 (写数据):
  bus  = data              总线换数据
  A0   = 1                 声明: 这是数据
  /WR  ↓ 再 ↑              写脉冲 (上升沿锁数据)
  → ADDR[n]=1 的 HC374 在 /WR↑ 锁住 data (A0=1, data_strobe 有效)
  → 其余 HC374 不变

/CS = 1                    ← 事务结束 (释放芯片)
```

### 2.3 核心设计原则：A0 只选择，/WR 才锁存

⚠️ **关键**（仿真逼出来的正确设计）：
- **A0 的跳变本身不触发任何锁存**——A0 只是"这次 /WR 脉冲锁到地址锁存器还是数据锁存器"的选择信号。
- **所有锁存动作只由 /WR 的上升沿触发**。
- 这就保证了寄存器绝不会因为 A0 切换而出现中间态。

早期错误方案（已弃用）：地址锁存用 HC373（透明，LE=NOT(A0)），A0 从 0→1 锁地址。问题：A0 跳变时 write_strobe=A0·/WR 也跳变，目标寄存器会**短暂锁到地址值**（如写 reg0=A5 时它先变成 0x01 再变成 0xA5）——数据乱跳，音频上是实打实的杂音。

**正确方案**（已验证）：地址锁存用 **HC374（边沿）**，地址和数据的锁存 CP 都从 /WR 上升沿派生，A0 做选择：
```
addr_cp     = NOT(/CS) AND NOT(A0) AND /WR     // 写地址拍 (/WR↑ 锁地址)
data_strobe = NOT(/CS) AND A0     AND /WR     // 写数据拍 (/WR↑ 锁数据)
reg_cp[n]   = ADDR[n] AND data_strobe          // 地址位一对一选通
```

### 2.4 仿真验证结果（2026-07-07，tb/psg3_bus_tb.v，0 错误 ✅）

```
✅ 写 reg0(地址 0x01) = 0xA5 → reg0=A5, reg1 保持 00
✅ 写 reg1(地址 0x02) = 0x3C → reg1=3C, reg0 保持 A5
✅ 覆盖 reg0 = 0xFF → reg0=FF, reg1 保持 3C
🎉 总线协议验证通过 (带 /CS, 无中间态)
```

**monitor 验证"无中间态"**：每次写寄存器，monitor 只捕获到**一次跳变**（直接变成最终数据值）。对比无 /CS 方案的两次跳变（先地址值再数据值），现在完全干净——这才是雅马哈协议该有的样子。

### 2.5 读时序（暂不实现）

YM2413 有 /RD 读寄存器。PSG 是只写设备（和 v0.1-v0.3 一致），/RD 暂不接。未来需要回读（如 SCC busy）再加。

### 2.6 关键约束

- **/CS 是事务边界**：整个两拍写期间 /CS=0，事务外 /CS=1。事务外（/CS=1）一切锁存无效，杜绝总线毛刺和 setBitMode 切换时的引脚跳变误触发（PSG2 v0.3 的老问题，/CS 自然解决）。
- **A0 只选择不锁存**：A0 决定 /WR 上升沿锁到地址还是数据，A0 自身的跳变不触发锁存。
- **/WR 上升沿是唯一锁存触发**：地址和数据锁存都靠它。
- **/CS 不能接 GND**：必须由 FT232H 控制（D7），因为 /CS 是事务保护壳，常有效就失去了消除中间态的作用。

## 三、硬件架构（地址一对一选通 + /CS 事务保护，已仿真验证）

### 3.1 核心数据通路（仿真验证通过 ✅，带 /CS 无中间态）

```
FT232H C0-C7 ──→ D0-D7 复用总线
                   │
       ┌───────────┴───────────┐
       │                        │
  ┌────▼─────────┐        ┌────▼──────────────────────────────┐
  │ U_ADDR       │        │  U_REG0..U_REG7 (8× HC374)         │
  │ HC374 (边沿!) │        │  每片 CP = ADDR[n] AND data_strobe│
  │ CP = addr_cp │        │  每片 D0-D7 = 复用总线             │
  └────┬─────────┘        │  Q = 寄存器输出 → 各通道功能       │
       │ ADDR[7:0] (独热码)└────┬──────────────────────────────┘
       │ 每位选通一个 374        │
       └────────────────────────┘

  控制逻辑 (关键, 消除中间态):
    cs_active  = NOT(/CS)                          // 事务期间=1
    addr_cp    = cs_active AND NOT(A0) AND /WR     // 写地址拍 /WR↑ 锁地址
    data_strobe= cs_active AND A0     AND /WR     // 写数据拍 /WR↑ 锁数据
    reg_cp[n]  = ADDR[n] AND data_strobe           // 地址位一对一选通

  ┌──────────────┐
  │ U_AND (选通)  │  AND 运算: cs·A0·/WR, cs·~A0·/WR, ADDR[n]·data_strobe
  │ HC08 ×N      │  门数核算见 3.4
  └──────────────┘

  ┌──────────────┐
  │ U_INV (非门)  │  NOT(/CS), NOT(A0)
  │ HC04 (2门)   │
  └──────────────┘
```

### 3.2 选通逻辑（仿真验证通过 ✅，0 错误 + 无中间态）

**核心公式**（tb/psg3_bus_tb.v 验证）：
```
cs_active   = NOT(/CS)
addr_cp     = cs_active AND NOT(A0) AND /WR     // 地址锁存 HC374 的 CP
data_strobe = cs_active AND A0     AND /WR     // 数据锁存公共触发
reg_cp[n]   = ADDR[n] AND data_strobe           // 每个数据 HC374 的 CP
```

**为什么无中间态**（仿真确认）：
- **A0 跳变不触发任何锁存**：addr_cp 和 data_strobe 都包含 /WR，A0 单独跳变时 /WR 不变，这两个信号虽有变化但不是上升沿（地址 HC374 是 CP 上升沿触发，电平变化无效）。
- **只有 /WR 上升沿才锁存**：写地址拍 /WR↑ 时 A0=0 → addr_cp 有上升沿 → 锁地址（data_strobe=cs·0·wr=0，无上升沿，数据 374 不锁）。写数据拍 /WR↑ 时 A0=1 → data_strobe 有上升沿 → ADDR[n]=1 的 374 锁数据（addr_cp=cs·1·wr，/WR 拉低时 addr_cp=0，拉高时 addr_cp=1 有上升沿... 等等）。

> ⚠️ **addr_cp 的细节**（仿真确认无害）：
> 写数据拍 /WR↑ 时，addr_cp = cs·NOT(A0)·/WR = cs·0·1 = 0（A0=1 时 NOT(A0)=0），无上升沿，地址锁存器不重锁。✅
> 写地址拍 /WR↑ 时，data_strobe = cs·A0·/WR = cs·0·1 = 0（A0=0），无上升沿，数据 374 不锁。✅
> 两者互斥，干净。

### 3.3 地址锁存器（⚠️ HC374 边沿，不是 HC373）

**关键修正**：地址锁存必须用 **HC374（边沿触发）**，不能用 HC373（透明）。
- HC373 透明锁存器在 LE 高时 Q 跟随 D，A0/bus 切换时有透明窗口，会漏数据。
- HC374 边沿触发，只在 CP 上升沿锁存，平时输出稳定，无竞争。

| HC374 脚 | 接 | 说明 |
|----------|---|------|
| D0-D7 | 复用总线 D0-D7 | 地址输入（独热码）|
| CP | **addr_cp = NOT(/CS) AND NOT(A0) AND /WR** | 写地址拍 /WR↑ 锁地址 |
| /OE | GND | 常输出 |
| Q0-Q7 | ADDR[0-7] | 8 位独热码地址 → 选通 8 个数据 374 |

> **vs 用户最初说的"地址用 373"**：用户原话"地址用 373 或 374"。仿真证明**必须用 374**（边沿），否则有透明窗口竞争。373 已弃用于地址锁存。

### 3.4 选通与门（门数核算）

需要的 AND 运算：
1. `cs_active = NOT(/CS)` — HC04 一门（反相，不是 AND）
2. `a0_n = NOT(A0)` — HC04 一门（反相）
3. `addr_cp = cs_active AND a0_n AND /WR` — 3 输入 AND = 2 个 2 输入 AND（HC08 两门级联）
4. `data_strobe = cs_active AND A0 AND /WR` — 3 输入 AND = 2 个 2 输入 AND（HC08 两门级联）
5. `reg_cp[n] = ADDR[n] AND data_strobe` × 8 — 8 个 2 输入 AND（HC08 八门）

**门数合计**：HC08 共 4+8 = 12 门 = **3 片 HC08**。HC04 用 2 门（NOT(/CS), NOT(A0)）= **1 片 HC04**。

| 芯片 | 用途 | 门数 | 片数 |
|------|------|------|------|
| HC08 #1 | addr_cp(2门) + data_strobe(2门) | 4 | 1 |
| HC08 #2 | reg_cp[0-3] = ADDR[0-3]·data_strobe | 4 | 1 |
| HC08 #3 | reg_cp[4-7] = ADDR[4-7]·data_strobe | 4 | 1 |
| HC04 | NOT(/CS), NOT(A0) | 2（余 4 备用）| 1 |

> **借门优化**：接口层独立用芯片，不借门给通道层（调试隔离原则）。HC04 余 4 门、HC08 无余门（12 门用满）。

> **⚠️ /OE 不能省 CP 选通逻辑**（2026-07-07 分析 + YM2413 驱动核对）：
> 曾设想用 HC374 的 /OE（输出使能/三态）挂 /WR 或 /CS 来省掉 CP 的地址选通与门。分析后不可行：
> - **HC374 锁存只由 CP 上升沿触发，/OE 只控 Q 输出三态（不控内部 q_reg 锁存）**。
> - 若所有数据 374 的 CP 直接连 /WR：地址拍 /WR↑ 时总线是地址值，所有 374 的 q_reg 被地址污染；/OE 让 Q 高阻也救不了——下次选中时输出的是被污染的错误值。
> - **CP 必须是 `ADDR[n] AND data_strobe`**（只有被选中的 374 在 /WR↑ 时锁存），8 个与门（U12+U13）省不掉。
>
> YM2413 真实协议核对（`Reference_Project/megagrrl-driver-revamp-2-20240226-saa/firmware/main/driver.c` Driver_FmOutopl3）：
> - /CS 在整个两拍事务期间常低（事务边界），不参与 CP 逻辑
> - /WR 地址拍/数据拍各脉冲一次，A0 区分
> - **专用芯片不用 /OE**（输出永远有效，三态是 74 门实现才有的概念）
>
> /OE 的真正价值在后续 SCC 阶段：寄存器 Q 要并到回读总线时，/OE 做三态隔离。PSG3 当前 reg 各走各的（不共用 Q 总线），/OE 全接 GND（常输出）。

## 四、芯片清单（总线层，不含通道功能）

### 4.1 总线控制芯片（必加，PSG3 相对 PSG2 v0.3 的新增）

| 位号 | 型号 | 功能 | 数量 |
|------|------|------|------|
| U_ADDR | **74HC374** | 地址锁存（8 位，**边沿**，CP=addr_cp）| 1 |
| U_AND1 | **74HC08** | addr_cp(2门) + data_strobe(2门) | 1 |
| U_AND2 | **74HC08** | reg_cp[0-3] = ADDR[0-3]·data_strobe | 1 |
| U_AND3 | **74HC08** | reg_cp[4-7] = ADDR[4-7]·data_strobe | 1 |
| U_INV | **74HC04** | NOT(/CS), NOT(A0)（用 2 门，余 4 门备用）| 1 |

> ⚠️ 地址锁存是 **HC374（边沿）不是 HC373**——仿真证明透明锁存器有竞争（见 3.3 节）。
> **借门优化可能**：HC04 剩 4 门、HC08 各片若有余门可借给通道（方波占空比链用 HC08）。接线表整体规划。

**总线层新增：5 片**（1×HC374 + 3×HC08 + 1×HC04）

### 4.2 数据寄存器（8× HC374，替代 PSG2 v0.3 的 2 片）

PSG2 v0.3 有 2 片 HC374（方波控 + 噪音控）。PSG3 扩到 8 片，每片一个寄存器。**地址用独热码**（见 2.1 节）：

| 位号 | 独热码地址 | 寄存器 | 用途（PSG3 初版规划） |
|------|----------|--------|---------------------|
| U_REG0 | 0x01 | CH0 period | 方波通道频率（8 bit）|
| U_REG1 | 0x02 | CH0 控制 | 方波：音量(4) + 占空比(2) + mode(1) + ref(1) |
| U_REG2 | 0x04 | 噪音控制 | 噪音：音量 + 频率挡 + 绑定 |
| U_REG3 | 0x08 | 预留 | 方波通道 2 period（未来）|
| U_REG4 | 0x10 | 预留 | 方波通道 2 控制（未来）|
| U_REG5 | 0x20 | 预留 | 锯齿/三角通道（design.md 5.5 规划）|
| U_REG6 | 0x40 | 预留 | — |
| U_REG7 | 0x80 | 预留 | SCC 波形表地址/数据窗口（未来）|

> **本版实现 reg0/reg1/reg2**：CH0 方波（reg0+reg1，复用 PSG2 v0.3 方波逻辑）+ 噪音（reg2，复用 rev.a）。reg3-7 预留，地址位悬空。

## 五、PSG3 通道规划（总线通了之后逐步加）

### 5.1 第一阶段：总线协议验证（CH0 方波 + 噪音）✅ 已完成

**目标**：把 PSG2 v0.3 的 CH0 方波 + CH1 噪音搬到 PSG3 总线上，验证 YM2413 协议跑通。

- U_REG0 = CH0 period（替代 PSG2 的 HC373 period 锁存）
- U_REG1 = CH0 控制（替代 PSG2 的方波 HC374）
- U_REG2 = 噪音控制（替代 PSG2 的噪音 HC374）

**通道硬件复用 PSG2 v0.3**：方波（2×HC161+HC00+HCT74+HC374+TLC7524+占空比链 HC74+HC08+HC153+HC4053）、噪音（rev.a：HC374 LFSR + CD4070 + HC161 + HC153 + HC374控 + TLC7524 + HC00）。这些通道逻辑全部不变，只是寄存器从"FT232H 直驱 D 端"改成"从总线 374 接收"。

**RTL**：`rtl/psg3_top.v`。**验证**：`tb/psg3_top_tb.v`，0 错误（见 7.1 节）。方波频率/占空比/静音、噪音出声、两通道同时工作互不干扰——全部通过。

### 5.2 第二阶段：扩到 2 方波通道

加 CH1（U_REG2 + U_REG3 + 一套方波通道硬件）。方波通道硬件复用，芯片数 = CH0 × 2。

### 5.3 第三阶段：锯齿/三角通道（design.md 5.5 节）

U_REG4 + U_REG5 + 锯齿通道硬件（2×HC161 + HC273 毛刺滤除 + 2×TLC7524 级联 + HC374 音量）。
三角用 CD4029 可逆计数 + CD4027 方向控制替换 HC161。

> **总线协议一次定终身**，通道功能分阶段加。这是 YM2413 复用的核心好处——地址空间预留够（8 位 = 256 寄存器），加通道不改总线。

## 六、与 PSG2 v0.3 的接线变化（迁移指南）

### 6.1 不变的部分（通道硬件）

- 方波通道：HC373→HC161×2→HC00→HCT74→HC374→TLC7524 + 占空比链（HC74/HC08/HC153/HC4053）
- 噪音通道 rev.a：HC374 LFSR + CD4070 + HC161 + HC153 + HC374控 + TLC7524 + HC00
- 电平转换层：LS373（FT232H 3.3V→5V，所有信号仍要过）
- 时钟：64kHz 全局晶振
- TLC7524 正向接法、输出链路（耦合电容+运放+喇叭）

### 6.2 改变的部分（控制总线）

| PSG2 v0.3 | PSG3 v0.4 |
|-----------|-----------|
| FT232H C0-C7 直连各 374/373 的 D 端 | C0-C7 → 复用总线 → 经 LS373 电平转换 → 总线（地址+数据共用）|
| D4=LE → HC373 period | D4=A0 → 总线协议控制（0=地址, 1=数据）|
| D5=A0 → 方波 HC374_CP | D5=/WR → 写脉冲（上升沿锁存）|
| D7=A1 → 噪音 HC374_CP | D7=/CS → **片选（事务边界，必须 FT232H 控制，不接 GND）**|
| period 走独立 HC373 | period 走 U_REG0（独热码地址 0x01 的 HC374）|
| 方波控走独立 HC374（D5 选通）| 方波控走 U_REG1（地址 0x02 的 HC374）|
| 噪音控走独立 HC374（D7 选通）| 噪音控走 U_REG2（地址 0x04 的 HC374）|

### 6.3 新增的部分（总线层）

- U_ADDR（**HC374** 地址锁存，边沿）
- U_AND1 + U_AND2 + U_AND3（3× HC08 选通与门）
- U_INV（HC04，NOT(/CS) + NOT(A0)）
- U_INV（HC04 或借门，NOT(A0)）

### 7.2 上板硬件踩坑（2026-07-08 实测，下个窗口必读）

PSG3 v0.4 上板阶段发现的问题，都不是仿真能预见的物理现象。后续版本务必遵守：

**1. 噪音 HC00 振荡器串扰方波 /PE（频率抖动根因）**
- 现象：方波频率异常抖动
- 根因：噪音防锁死振荡器（HC00 门3 + 2×0.1μF，MHz 级宽带噪声）和方波通道**共用同一片 HC00**，芯片内部串扰让方波 /PE（reload 核心）受干扰
- 修复：**噪音防锁死振荡器必须用独立 HC00，不能和方波通道共芯片**
- 教训：振荡器/噪声源类电路对数字逻辑通道是干扰源，物理隔离（独立芯片）是刚需

**2. 噪音绑定 tc_hi 反向干扰方波**
- 现象：噪音开绑定模式后方波又受干扰
- 根因：方波 tc_hi 直连噪音绑定 2 选 1（HC00 门3），噪音侧电路回灌影响方波信号源
- 修复：**tc_hi 经一级 HC00 反相器（缓冲隔离）后再接噪音通道**
- 教训：**噪音通道的电路会回灌干扰方波信号源**——所有跨通道信号（尤其方波→噪音）都要做单向隔离（反相器/缓冲器）。这是 PSG3 上板发现的第二条同类规律（和振荡器串扰同源）

**3. 占空比分频链必须用 HCT74（国产 HC74 toggle 失效，第三次踩坑）**
- 现象：占空比调整完全失效（只有一挡出声，其余无声）
- 根因：占空比分频链 U24（HC74）做 toggle ÷2，国产 HC74 在该翻转频率下 Q2/Q3 不动作 → 恒定 → duty_25/duty_125/duty_25f4 全是直流
- 修复：**U24 换 74HCT74**（和 U23 sync+toggle 一样）
- 教训：**所有 HC74 做 toggle/分频的场合都必须用 HCT74，不能用国产 HC74**。项目第三次踩同一个坑（v0.1 toggle、PSG3 占空比链）。CLAUDE.md 已记录，但每次接新 HC74 还是容易忘

**4. 驱动写入必须 MPSSE 批量打包（不能逐条发 + sleep）**
- 现象：ADSR 音乐慢 + 卡顿 + 方波毛刺；噪音独立频率只有 2 档有用
- 根因：`_bus_write` 逐条发 MPSSE 命令 + 每条 2ms sleep，一次写入 ~20ms（远超 ADSR TICK=5ms）；写入慢导致锁存建立时间不足/数据未稳定
- 修复：**整个事务的 D 口/C 口操作打包成一个 `bytes`，一次 `dev.write` 发完**（FT232H MPSSE μs 级顺序执行）
- 教训：v0.3 的逐条发 + 2ms sleep 在 v0.4 两拍写协议下完全行不通（命令数翻倍）。YM2413 两拍写必须批量打包

**5. _bus_write 时序：A0/CS 跳变必须在 WR=0 期间**
- 现象：方波频率抖动 + 占空比失效（驱动层面）
- 根因：`addr_cp = cs&a0_n&wr`，`data_strobe = cs&A0&wr`——A0 或 CS 在 WR=1 时跳变，会让 strobe 产生 0→1 上升沿，锁到错误值（总线此时还是上一个数据）
- 修复：**所有 A0/CS 跳变都在 WR=0 期间进行**（此时两 strobe 都=0，跳变不锁存），WR 上升沿只在 A0/CS 稳定后给出
- 教训：组合逻辑选通（strobe = 多信号 AND）对跳变顺序敏感，A0/CS 这种"选择信号"的跳变时机必须避开锁存上升沿

### 7.3 VGM 式预录制 + 噪音乐器（鼓组，2026-07-08）

PSG3 v0.4 给 ADSR 播放器加了**噪音乐器（鼓组）** + **VGM 式预录制回放**。核心思路参照 VGM（Video Game Music）格式：播放前把整首曲子 dump 成 `[(time_ms, port, data)]` 事件流，回放就是按时间戳顺序写寄存器，无运行时调度。

**噪音乐器 ini 定义**（`[noise_N]` 块）：
```ini
[noise_1]
noise_ins_name = bd              乐器名 (song notes 里用此名当音级, 如 bd:1)
noise_volume = 15,13,10          音量包络 (逐步衰减)
noise_step = 0.1s                包络步进时长 (key_on 后序列步进, 不影响音长)
noise_mode = independent          independent(独立分频) / bind(绑定方波 TC)
noise_freq = /16                  频率挡: /2 /4 /8 /16
```

**song 里使用噪音乐器**（方波 + 噪音分行，并行播放）：
```ini
[song_8]
notes = 1:1 2:1 3:1 1:1 | 1:1 2:1 3:1 1:1 |     ← 方波旋律行
        bd:1 sd:1 bd:1 sd:1 | bd:1 sd:1 bd:1 sd:1 | ← 鼓点行
```

**音长 = beats × beat_t（统一标准）**：
- 方波音长和噪音音长都用 `beats × beat_t`，两者严格等长对齐
- ADSR 包络（A/D/S/R）是**音色形状**，不影响音长——release 在音长内 tick 完，不额外加时
- 噪音 `noise_step` 是 key_on 后包络序列的步进，音长到直接 key_off 切断（序列没跑完也切）

**VGM 预录制流程**（`dump_song`）：
1. **双轨并行时间轴**：方波轨和噪音轨各自从 0 累计时间（方波行用方波轨累计，噪音行用噪音轨累计），两轨并行不串行
2. 方波轨：用 Voice 跑 ADSR 模拟，Recorder 拦截写入转事件（只记变化点）
3. 噪音轨：展开 NoiseInstrument 包络（按 step 逐步输出 vol，音长到切断）
4. 合并排序 → `[(time_ms, port, data)]`，port 区分 sq_period/sq_ctrl/noise

**回放**（`play_song`）：按时间戳顺序写硬件（`psg.set_period`/`_bus_write(0x02, ctrl)`/`_bus_write(0x04, noise_ctrl)`），到点写、没到点 sleep。切歌/退出立即中断；速度/音色/颤音变化不中断当前曲，**下一曲 dump 时生效**（重新读当前 tempo/tone/vibrato）。

**实时乐谱打印**：
- 有噪音乐器时：一次性打印两行（方波行 + 鼓点行），完整预览
- 纯方波时：实时单行打印（边播边打 `[token]`，v0.3 老方式）

**寄存器地址**（独热码，和驱动一致）：reg0(0x01)=方波period, reg1(0x02)=方波控制, reg2(0x04)=噪音控制。
