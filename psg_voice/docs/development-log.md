## 1. 项目背景与目标

在 74HC-Chiptune 项目（纯 74HC 逻辑合成器）框架下，新建一个**单音 PSG 子项目**，作为：
- 工具链验证（iverilog + vvp 能否编译仿真 74HC 模型）
- 最简方波音源 demo（面包板可快速搭建）
- 后续多通道/波表合成的基础模块

**核心约束**：音源内部**全 74HC 实例化**，接口/testbench 可用抽象 Verilog。

## 2. 需求演进（关键决策记录）

需求经过多轮收敛，每轮都伴随架构重新评估：

| 轮次 | 需求 | 推导的方案 | 结果 |
|------|------|-----------|------|
| 初版 | 88 键钢琴音区(A0~C8) + gate | 18-bit 计数器，11+ 片 | ❌ 过度设计 |
| 第2轮 | 参考真实 PSG 位宽 | 查 AY8910(12bit)/SN76489(10bit) | 发现"period 计数翻转"机制 |
| 第3轮 | 精简 76 键 + 2MHz 时钟 | ÷16 预分频 + 11-bit period | 仍需 6 片 |
| 第4轮 | **demo 项目，覆盖常用音即可** | 8-bit period + 外部慢时钟 | ✅ 定稿基础 |
| 第5轮 | **压到 5 片以内** | HC04+HC08 合并为 HC00 | ✅ 最终 5 片 |

### 关键转折：参考 STC 工程的真实 PSG

在 `E:\working\vscode-projects\STC_Chiptune\STC32G144K246\` 读到 `ay8910.c` / `sn76489.c`，发现经典 PSG 的核心机制：

```c
// STC ay8910.c:164 / sn76489.c:110
count += incr;
if (count >= period) { toggle; count -= period; }
```

**这不是相位累加器，而是"计数到 period 就翻转"** —— 和最初设想的 prog_timer 架构一致。SN76489 真实参数：
- SN_CLK = 3,579,545 Hz，÷16 预分频 → 223.7kHz 采样
- period 10-bit

这奠定了用 **HC161 的 TC（计满标志）+ 同步预置** 代替独立比较器（HC85）的思路。

## 3. 参数推导（参考真实 PSG）

### 频率公式
```
f = clk / (2 × (256 - period))
```
（计数器从 period 自增到 0xFF，共 256-period 步，到顶翻转）

### @125kHz 外部慢时钟的覆盖范围

| period | 频率 | 对应音 |
|--------|------|--------|
| 0 | 244 Hz | ≈ B3 |
| 114 | 440 Hz | **A4（标准音）** |
| 185 | 880 Hz | **A5** |
| 255 | 62.5 kHz | 超声波（上限）|

→ 覆盖 C4~C6 常用旋律音区，demo 足够。

### 为什么不用 2MHz 直接计数
- 2MHz + 8-bit → 只能发高音（>3.9kHz），覆盖不了 A4
- 2MHz + ÷16 预分频 → 244Hz~62kHz，但多 1 片 HC161
- **结论：用外部 125kHz 慢时钟直接进 clk，省掉预分频芯片**

## 4. RTL 设计与迭代

### 最终架构（5 片 74HC）

```
                    ┌─────────┐
  period_in[7:0] ──→│ HC373   │── period_q[7:0] ──┐
        /LE ────────→│ (锁存)  │                   │
                    └─────────┘                   ↓
                    ┌─────────┐   count    ┌─────────┐
              clk ─→│ HC161   │──────────→│ (观测)  │
                    │ HC161   │── TC_HI ──┼─────────┤
                    └─────────┘           │         │
                          ↑               ↓         ↓
                          │           ┌─────────┐  ┌─────────┐
                          └──/PE──────│ HC00    │←─│ HC74    │
                                      │ (与非)  │  │ (D触发器)│
                          gate ──────→│         │←── TOGGLE_Q
                                      └────┬────┘
                                           ↓
                                       PSG_OUT → 蜂鸣器
```

### 工作流程
1. host 写 period 到 HC373（LE 高→低脉冲锁存）
2. 两片 HC161 级联计数，从 period 数到 0xFF
3. TC_HI=1 → 经 HC00 反相得到 /PE=0 → 下个 clk 同步预置回 period（重装）
4. TC_HI 同时进 HC74 半片（sync）用 clk 同步消毛刺 → RELOAD_PULSE
5. RELOAD_PULSE 上升沿触发 HC74 另半片（T 触发器）翻转 → TOGGLE_Q
6. TOGGLE_Q & gate 经 HC00 两级与非 → PSG_OUT 方波

## 5. 芯片优化（6 片 → 5 片）

### 优化前的 6 片方案
| # | 型号 | 职能 |
|---|------|------|
| 1 | HC373 | period 锁存（透明锁存器，替代 HC377） |
| 2,3 | HC161×2 | 计数器 |
| 4 | **HC04** | PE 反相 |
| 5 | HC74 | 同步 + T 翻转 |
| 6 | **HC08** | gate 与门 |

### 用户提出合并建议
> "hc04 和 hc08 能合并吧，用 hc02 或者 00 就可以吸收掉那个 gate。"

### 德摩根律推导
gate 与门逻辑：`wave_out = gate & toggle_q`

用与非门实现正与：`wave_out = ~(~(gate & toggle_q))`（两级与非）

HC00 一片 4 路：
- **第 1 路**：PE 反相（输入短接当反相器）`pe_n = ~(tc_hi & tc_hi)`
- **第 2 路**：gate 与门第一级 `gate_nand1 = ~(gate & toggle_q)`
- **第 3 路**：gate 与门第二级 `wave_out = ~(gate_nand1 & gate_nand1)`
- **第 4 路**：闲置

→ **HC04 + HC08 合并为 1 片 HC00，总 5 片**。仿真验证 A4=440Hz / A5=880Hz 精确。

> 注：HC02（或非门）也能合并，但需 toggle 用 Q_n（反相输出）。选 HC00 保留 toggle 用 Q，逻辑更直观。

## 6. 仿真验证与 Bug 修复

### 工具链
- iverilog 14.0 (devel) + vvp（来自 oss-cad-suite）
- 路径：`D:\Program Files\oss-cad-suite\bin`

### Bug 1：组合反馈导致死锁（初版设计）
**现象**：初版用 `match` 异步清零计数器，形成组合反馈，PSG 卡死不振荡。
**修复**：改用 HC161 的**同步预置**（PE）重装，避免异步清零反馈。

### Bug 2：TC 毛刺导致 T 触发器多触发（核心 Bug）
**现象**：A4 频率正确(440Hz)，但 A5 实测 1320Hz（应为 880Hz），恰好是 1.5 倍。

**定位过程**（`psg_debug_tb.v`）：
```
2840 clk 内（期望 40 个重装周期）
tc_hi 上升沿: 40   ← 正确，TC 无毛刺
toggle 上升沿: 30  ← 应为 20！T 触发器多翻了
```

**根因**：HC161 的 TC 是组合输出（`CET & Q==0xFF`），count 经历 0xFF 转换瞬间有毛刺，直接当 HC74 时钟会多触发。

**修复**：用 HC74 的**半片做 D 触发器**，把 TC 用 clk 同步一拍（RELOAD_PULSE）后再驱动 T 触发器。修复后实测频率完全精确。

> 这是真硬件也会遇到的问题，面包板搭的时候必须保留这级同步。

### Bug 3：PE 极性需要反相
**现象**：做接线表时发现 RTL 用 `pe_n = ~tc_hi`（Verilog `~`），但硬件清单没有反相器。
**验证**：试过 PE 直接接 TC（不反相）→ 仿真 0Hz 死锁（count 卡在 0x00 不停被预置）。
**修复**：必须反相。用 HC00 第 1 路实现（后来合并方案里）。

### 变更 4：HC377 → HC373（库存替代）
**起因**：查库存无 HC377，但有 HC373 和 HC374。

**选型分析**：
| 芯片 | 类型 | 使能方式 | 接 clk |
|------|------|---------|--------|
| HC377（原）| 边沿 D 触发器 | /Enable（低有效）+ CLK 沿 | ✅ |
| HC373（选中）| **透明锁存器** | **LE（高透明/低锁存）** | ❌ 不接 |
| HC273 | 边沿 D 触发器 | 无使能（每 CLK 沿都锁存）| ✅ |

HC273 没有使能端（每 clk 都锁存 D），需要 MCU 保证 D 端稳定，复杂。
HC373 是电平敏感锁存：LE=1 时 Q 跟随 D，LE=0 时锁存。**MCU 直接控制 LE，与 PSG 的 clk 完全解耦**，反而比 HC377 更简单。

**RTL 变更**：
- `hc377 u_period(.Enable_bar, .D, .Clk, .Q)` → `hc373 u_period(.OE_n(1'b0), .LE, .D, .Q)`
- 接口信号 `period_we`（低有效）→ `period_le`（高透明/低锁存），极性反
- U1 不再接 clk

**写时序简化**（HC377 → HC373）：
```
HC377: MCU_D=值; WE↓; 等10μs(覆盖1个clk沿); WE↑   ← 依赖 clk 相位
HC373: MCU_D=值; LE↑(透明); 等1μs; LE↓(锁存)      ← 不依赖 clk, 任意时刻可锁
```

**仿真验证**：A4=440Hz / A5=880Hz 精确，gate 正常，MCU 接口验证通过。

### 变更 5：控制器选定 FT232H（PC USB 桥接）
**起因**：不用写 MCU 固件，直接 PC 上跑 Python 控制 PSG。

**驱动方案探索**：
- pyftdi（libusb 后端）：需 Zadig 换驱动 + 放 libusb-1.0.dll，麻烦
- **ftd2xx（FTDI 官方 D2XX）**：系统已自带 FTD2XX.dll，`pip install ftd2xx` 即用，**零折腾选中**

**MPSSE GPIO 方案**：
- ACBUS C0-C7（8 个）→ period_in[7:0]
- ADBus D4-D7（4 个）→ LE / GATE / RST / 备用
- 合计 12 GPIO，PSG 需 11，够用

**电平注意**：FT232H 是 3.3V 输出。驱动 74HCT（VIH=2.0V）OK；驱动 74HC（VIH=3.5V）可能不够。本项目用 HCT 系列，兼容。

**可用性验证**（`test_ftdi.py` / `test_ftdi2.py` / `test_ftdi3.py`）：
- 硬件识别：`Single RS232-HS`（FT232H 典型名）
- FTD2XX.dll 加载成功，设备能打开、能 reset
- bitbang 模式可切、**MPSSE 模式可切**（关键）
- **GPIO 自检 12/12 全通过**（`test_gpio_selfcheck.py`，写高读高、写低读低）

### 最终验证结果
```
A4 目标 440Hz → 实测 440Hz（44 上升沿/12500 clk）✓
A5 目标 880Hz → 实测 880Hz（88 上升沿/12500 clk）✓
Gate OFF → 0 上升沿（静音正常）✓
Gate ON  → 发声正常 ✓
MCU 异步 WE 写入正确，音高实时可切 ✓
FT232H GPIO 12/12 自检通过 ✓
```

## 8. MCU 接口规范

详见 [`docs/mcu-interface.md`](mcu-interface.md)。**并行直驱，0 额外芯片，11 个 GPIO**。

核心时序：MCU 把 period 放到 D[7:0]，LE 拉高（HC373 透明）→ 保持 → LE 拉低（锁存）。HC373 不接 clk，与 PSG 振荡解耦，任意时刻锁存有效。

特性：写一次 PERIOD → PSG 自激振荡持续发声，MCU 可休眠，只在换音/开关时动 GPIO。

## 10. 硬件调试实录（上电踩坑全记录）

> 仿真全过、网表审查全过，但**上电后没有任何声音**。以下是从"完全不响"到"完美发声"的完整调试过程，每个 bug 都是真实踩到的。

### 调试起点：现象描述

上电后跑 C 大调音阶脚本，**只听到"bo bo"敲击音**（每次写 period 瞬间响一下），音高随 period 变化，但**没有持续方波音**。

### Bug A：3.3V 电平不够（最大根因）⭐⭐⭐

**现象**：74HC161 自由计数（Q0-Q7 恒定八度倍频），完全不响应 D0-D7 写入的 period。

**排查过程**：
1. 量 74HC00 的 tc_hi（Pin 1/2）和 pe_n（Pin 3）：都有低频脉冲 ✓
2. 量 74HC161 的 PE（Pin 9）：和 pe_n 同款脉冲 ✓
3. 但 161 就是不重装，自由跑
4. **关键测试**：把 74HC373 的 LE（Pin 11）**直接飞线接 VCC（5V）** → period 数据突然正常，161 开始响应 D0-D7，Q3 出方波音阶！

**根因**：FT232H 的 GPIO 输出是 **3.3V**，而 74HC 的 VIH = VCC×0.7 = **3.5V**。3.3V 被判成低电平 → LE 永远"低" → 373 一直锁存初始值 0 → period 进不去。
- "敲击音" = FT232H 写脉冲的瞬时电平跳变 transient，不是真实方波
- 3.3V 量得到、有咔哒音，但逻辑上不够高，极隐蔽

**修复**：加 **74LS373 电平转换层**（VIH=2.0V 能识别 3.3V，5V 供电输出 5V）。详见接线表 U0。

### Bug B：rst_n 的 3.3V 让 74HC74 锁死

**现象**：74HC74 的 reload_pulse（Q1）恒定 0V，toggle 不翻转。

**排查过程**：
1. 量 74HC74 输入：D1（tc_hi）有脉冲 ✓、CLK1（clk）有方波 ✓、PRE1=4.8V ✓、电源 ✓
2. 量 CLR1（Pin 1）= **3.2V**（来自 FT232H C2 的 rst_n）
3. 3.2V < VIH 3.5V → CLR 被判低 → 持续异步清零 → Q1 恒 0

**根因**：同 Bug A，rst_n 也是 3.3V 电平不够。

**修复**：rst_n 过 LS373 转换成 5V 后，CLR1 收到 4.8V，74HC74 正常工作。

### Bug C：74HC74 芯片损坏

**现象**：Bug B 修复后（CLR 接 VCC），reload_pulse **仍然恒定 0V**。

**排查过程**：
1. 所有输入确认正常（D1/CLK1/PRE1/CLR1/电源）
2. 量 Q1（Pin 5）= 0V，Q1_n（Pin 6）= 4.8V，互为反相但都锁死
3. 输入全对却无响应 → 该半片损坏

**修复**：**换一颗 74HC74**，toggle_q 立即输出纯净方波。

> 教训：硬件调试时，逻辑全对仍不工作，要考虑芯片本身损坏的可能（ESD/过流/出厂不良）。

### Bug D：gate 同样被 3.3V 卡住

**现象**：toggle_q 有方波了，但 wave_out 还是没声音。

**排查过程**：
1. toggle_q（74HC00 Pin 5）有方波 ✓
2. gate（74HC00 Pin 4）= FT232H C1 输出 = **3.3V** ⚠️
3. 74HC00 的 VIH=3.5V，3.3V 的 gate 被判低 → `gate & toggle_q` 恒 0 → 静音

**根因**：同 Bug A/B，gate 也是 3.3V 电平不够。

**修复**：gate 过 LS373 转换成 5V，wave_out 输出正常方波。

### 调试终点：完美发声 🎉

四个 bug 全部定位修复后：
- ✅ period 响应 D0-D7（LS373 转换 LE/数据）
- ✅ 161 计数 + 重装正常
- ✅ toggle_q 纯净方波（换 74HC74 + rst 转换）
- ✅ wave_out 输出（gate 转换）
- ✅ C 大调音阶清晰发声，音高准确

### Bug E：MPSSE 下 D1/D2 不可靠（period 数据被篡改）⭐

**现象**：PSG 通了但"听起来在一个八度内循环"，音高变化不明显。

**排查过程**：
1. 写不同 period，回读 FT232H 的 D 口 → 发现写入值和回读值对不上
2. 逐位测试：写 D1=0 读到 1，写 D2=0 读到 1 → **D1/D2 被强制拉高**
3. D1/D2 是 MPSSE 的 **TDI/TDO** 引脚，MPSSE 模式下被 SPI 引擎占用，0x80 命令写的值会被干扰

**根因**：FT232H 在 MPSSE 模式下，ADBUS 的 D1(TDI)/D2(TDO) 不能可靠地当 GPIO 用，写 0 会被强制拉成 1，导致 period 的 bit1/bit2 永远是 1，实际 period 值被篡改。

**修复**：**period 数据改用 ACBUS C0-C7**（0x82 命令，完全可控），控制信号（LE/GATE/RST）挪到 ADBus D4/D5/D6（这几个验证可靠）。

> **教训**：FT232H MPSSE 模式下，ADBUS D0-D3 是 SPI 专用（TCK/TDI/TDO/TMS），**不能当 GPIO**。能用的 GPIO 是 **ACBUS C0-C7（8 个）+ ADBus D4-D7（4 个）**。

### Bug F：时钟频率选择（8-bit period 的音区矛盾）⭐

**现象**：扫频时发现音区覆盖和高音音准存在根本矛盾。

**排查过程**：测试三个时钟频率，对比音区覆盖和高音区 period 分辨率：

| 时钟 | 最低音 | C5 period/步数 | C7 period/步数 | 问题 |
|------|--------|---------------|---------------|------|
| 125kHz | B3(244Hz)，C3 到不了 | 137/119 | 226/30 | 低音覆盖不足 |
| 16.7kHz | C1(32.7Hz) | 240/16 | 252/4 | **C5 步数太少→方波消失** |
| **64kHz** | **C3(130.8Hz)** | **195/61** | **241/15** | **甜点！平衡** |

**根因**：8-bit period 的频率公式 `f = clk/(2×(256-period))`，period 范围 0~255。低音需要大 period（→需要低 clk），高音需要 period 远离 255（→需要高 clk 保证步数）。两者矛盾，**8-bit 无法同时覆盖很宽音区且保持高音音准**。

**关键发现**：高音区"没声音"的根因是 **period 步数太少**（如 16.7kHz 时 C5 只有 16 步，C7 只有 4 步），计数器重装逻辑在极少步数时不稳定，方波消失。

**解决**：选 **64kHz** 作为甜点时钟，覆盖 C3~C8（5 个八度），各音区步数充足。

### Bug G：74HC74 高频翻转失效（需换 74HCT74）⭐⭐

**现象**：64kHz 下扫频，C3~C5 正常，但 **A6(1760Hz) 以上 toggle_q 消失**，高音区没声音。

**排查过程**：
1. 量 tc_hi：高音区正常（窄脉冲）
2. 直接把 toggle_q 接 tc_hi：能出声但带谐波（占空比 ~3% 的窄脉冲，频谱宽）→ 证明 tc_hi 正常，问题在 74HC74
3. 加去耦电容：无效
4. **换 74HCT74：A6+ 全部恢复正常**

**根因**：74HC74 在面包板环境（寄生电容大、飞线长）下，高频翻转能力不足。A6=1760Hz 对标称 f_max(20MHz+) 来说本不该是问题，但实际受工艺/个体/寄生参数影响，HC 系列在高频翻转时表现差。

**对比**：
| 系列 | A6(1760Hz) | 说明 |
|------|-----------|------|
| 74HC74 | ❌ 失效 | 面包板高频翻转不行 |
| 74HCT74 | ✅ 正常 | TTL 工艺，翻转更快，余量大 |

**解决**：toggle 触发器**必须用 74HCT74**（或 74AC74），不能用 74HC74。

> **教训**：高频翻转/窄脉冲采样的场合（toggle、sync D 触发器），**优先用 HCT/AC 系列，不要用 HC**。HC 系列虽然标称 f_max 高，但实际在面包板/寄生电容环境下表现远不如 HCT。
>
> 这是本项目第二次因 74HC74 出问题（第一次是芯片损坏，第二次是高频失效），说明 HC74 在这个应用里就是个雷。

### 调试方法论总结

1. **从输出端往回追**：没声音 → 量 wave_out → 没方波 → 量 toggle_q → 没翻转 → 量 reload_pulse → 锁死 → 量输入 → 找到根因
2. **对比法**：74HC161 和 74HC74 的 CLR 共用 rst_n，161 工作正常 = rst_n 信号"看起来"没问题（但实际电平不够，只是 161 容忍度更高）
3. **飞线接 VCC 测试**：怀疑某信号电平不够时，直接飞线接 VCC（5V）绕过原信号源，立刻能判断是"信号问题"还是"芯片问题"
4. **仿真≠硬件**：3.3V 电平问题在仿真里完全体现不出来（仿真不模拟 VIH 阈值），必须实测
5. **信号回读验证**：怀疑数据线问题时，写值后回读 FT232H GPIO 状态，能立刻发现 D1/D2 篡改类问题
6. **换芯片是有效手段**：逻辑全对仍不工作，果断换芯片。本项目换了 2 次 74HC74（损坏 + 高频失效），最终换 HCT74 解决
7. **窄脉冲听感 = 谐波多**：直接听 tc_hi 是"带谐波的声音"，说明 tc_hi 是窄脉冲（占空比小），需经 toggle 转 50% 方波才是纯音

## 13. 经验总结

### 设计层面
1. **先查经典实现**：参考 SN76489/AY8910 真实 PSG 后，才发现"period 计数翻转"远比"相位累加器"省芯片。不要凭空设计。
2. **物理约束优先**："可编程 + 自激振荡"必须同时有 period 存储 + 终点检测 + 翻转。想省其中一个，会引入更复杂的问题。
3. **最小化思维**：HC161 自带 TC 就能当终点检测，不必用 HC85 比较器。HC04+HC08 能用德摩根律合并成 HC00。

### 仿真层面
4. **组合输出毛刺是真 bug**：HC161 的 TC 是组合逻辑，直接当时钟会多触发。必须用时序逻辑（D 触发器）同步一拍。这类问题只有仿真能发现，看 RTL 看不出来。
5. **测量窗口要避开切换点**：测频率时若窗口跨了 period 切换，会得到错误平均值（曾误判 A5=870Hz）。
6. **debug testbench 很有价值**：`psg_debug_tb.v` 打印 count 序列和 TC/toggle 上升沿计数，是定位毛刺 bug 的关键。

### 硬件层面
7. **Verilog `~` 不是硬件**：RTL 里的取反、与门等运算必须映射到真实芯片（HC00/HC04），否则原理图无法对应。做接线表时逼出了这个隐藏问题。
8. **网表审查比看图可靠**：原理图图像分辨率有限，但 PADS 网表给出精确的 pin-to-pin 连接，逐条核对最准。
9. **多查引脚定义**：74HC74 的 Pin10 是 PRE2 不是 CLK，这种张冠李戴只有对照 datasheet 才能发现。
10. **⚠️ 电平兼容性是最隐蔽的坑**（本次最大教训）：3.3V 控制器（FT232H）驱动 5V 74HC 系统，**必须过 TTL 电平器件（LS/HCT）转换**。3.3V 信号"量得到、有 transient（咔哒音）"，但 < VIH(3.5V) 会被判低，导致系统静默。仿真完全不体现这个问题。
11. **飞线接 VCC 是最快的诊断手段**：怀疑某信号电平不够时，直接飞线接 VCC 绕过原信号源——能工作 = 信号问题，不能工作 = 芯片问题。本次靠这个方法快速区分了"LE 电平不够"和"373 损坏"。
12. **芯片可能就是坏的**：逻辑全对仍不工作，要考虑芯片损坏。本次 74HC74 sync 半片损坏，换了就好。ESD/过流/出厂不良都可能。
13. **从输出端往回追**：没声音 → wave_out → toggle_q → reload_pulse → 输入。逐级量，定位最快。

### 流程层面
14. **工具链先行验证**：用 prog_timer（最简 4 片定时器）先验证 iverilog+vvp 能跑，再动手做复杂 PSG，避免工具问题干扰设计验证。
15. **仿真验证必要但不充分**：仿真过了 ≠ 硬件能工作。电平、毛刺、芯片损坏、接触不良都是仿真覆盖不到的。硬件调试阶段要有耐心，逐信号量测。
