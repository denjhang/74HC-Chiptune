# SCC 专用机架构决策（CPU + 混音协处理器）

> 决策时间：2026-07-05
> 决策依据：scc-benchmark.md（580 拍/采样实测）+ 8MHz 主频硬约束
> 这是 Glowworm-1 项目的转折点：从"研究萤火虫架构"转向"设计专用音源芯片"

## 一、决策三要素（已拍板）

| 维度 | 决策 | 理由 |
|------|------|------|
| **架构哲学** | CPU + 混音协处理器 | 保留可编程性（glowcc）+ 8MHz 下接近专用机性能 |
| **目标主频** | 直插 8MHz | 不堆主频，靠硬件加速填性能缺口 |
| **通道数** | 5 通道完整 SCC | 一步到位，分时复用硬件控制片数 |

## 二、为什么不选另外两条哲学

### ❌ 哲学 1：纯 CPU 主导（萤火虫 + 专用指令）

加专用指令（相位累加/查表/乘法封装为单指令）仍是 8 位序列化：
- 单通道混音最好估计 ~250 拍（理论，待验证）
- 5 通道 8MHz = 6400 Hz，**仍不够 17640 Hz**
- 8 位 ALU 是根本瓶颈，软件无解

### ❌ 哲学 2：硬件主导专用 SCC（CPU 退化）

- 5 通道独立硬件 = 35-45 片，超预算
- 抛弃 glowcc，自己写状态机，工作量大
- 不可编程，失去"特化但仍通用"的项目价值

### ✅ 哲学 3：CPU + 混音协处理器（最终选择）

- CPU 跑 C（ADSR/序列/UI），协处理器跑 5 通道混音
- **真并行**：CPU 发 START 后协处理器自主跑，CPU 同时做别的
- 8MHz 下每采样 453 拍预算，协处理器吃 ~100 拍，CPU 还剩 ~350 拍
- 片数 ~28 片（核心 ~22），落在 30-50 片预算内

## 三、系统架构

```
┌──────────────────────────────────────────────────┐
│  萤火虫 CPU 核（极简版，8MHz）                    │
│  - 砍：long long / double / IO1 / 大地址空间      │
│  - 留：基本整数运算、查表 ALU、RF、RAM、IO0       │
│  - 跑：main 循环 / ADSR / 歌曲序列 / 参数管理     │
└──────────┬───────────────────────────────────────┘
           │  专用指令接口：
           │    MIX_START  → 启动协处理器
           │    MIX_DONE?  → 轮询完成标志
           │    MIX_OUT    → 读混音结果
           ▼
┌──────────────────────────────────────────────────┐
│  SCC 混音协处理器（自主分时 5 通道）              │
│                                                   │
│  ┌─────────────┐                                  │
│  │ 通道计数器  │ 0 → 1 → 2 → 3 → 4 → done        │
│  │ HC161 (3位) │                                  │
│  └──────┬──────┘                                  │
│         │ 选通道                                   │
│         ▼                                          │
│  ┌─────────────────────────────────────────┐     │
│  │  每通道执行（硬件状态机，~10 拍/通道）：  │     │
│  │                                          │     │
│  │  1. phase[ch] += step[ch]                │     │
│  │     ├ HC161×2 (16位相位累加器)           │     │
│  │     └ step[ch] 来自 RF/RAM 寄存器        │     │
│  │                                          │     │
│  │  2. offs = phase.hi & 0x1F               │     │
│  │     └ HC08 (AND 掩码)                    │     │
│  │                                          │     │
│  │  3. sample = wave[ch][offs]              │     │
│  │     ├ 地址 = {ch[2:0], offs[4:0]}        │     │
│  │     └ 62256 SRAM (32B×5ch = 160B)        │     │
│  │                                          │     │
│  │  4. out = (sample * vol[ch]) >> 4        │     │
│  │     ├ 乘法 ROM 查表 (8×8→8 位带符号)     │     │
│  │     └ vol[ch] 来自寄存器                 │     │
│  │                                          │     │
│  │  5. mix += out                           │     │
│  │     └ HC161 累加器（11 位，5×127<2048）  │     │
│  └─────────────────────────────────────────┘     │
│                                                   │
│  循环 5 次后：输出 mix，置 MIX_DONE               │
└──────────────────────────────────────────────────┘
           │
           ▼  输出采样到 DAC → 喇叭
```

## 四、CPU 视角（API 设计）

CPU 用一个"命令寄存器 + 数据寄存器"组操作协处理器：

```c
// 协处理器寄存器（CPU 视角，内存映射或专用 IO）
extern u8 MIX_CMD;     // 写 1 = START，读 bit0 = DONE
extern u8 MIX_OUT;     // 混音结果（中心 128）
extern u8 MIX_CH;      // 通道参数索引（设 step/vol 时用）
extern u8 MIX_STEP_L;  // 写 step 低字节
extern u8 MIX_STEP_H;  // 写 step 高字节
extern u8 MIX_VOL;     // 写音量
extern u8 MIX_WAVE;    // 写波形数据（按 ch+offs 顺序）

// CPU 一个采样周期的工作（~30 拍 + 等待）：
void scc_tick() {
    MIX_CMD = 1;           // 启动协处理器（5 拍）
    while (!(MIX_CMD & 1));// 等待（协处理器跑 ~50 拍，CPU 阻塞）
    DAC = MIX_OUT;         // 输出（5 拍）
}

// 参数变化（按需，不在每采样执行）：
void scc_set_param(u8 ch, u16 step, u8 vol) {
    MIX_CH = ch;
    MIX_STEP_L = step & 0xFF;
    MIX_STEP_H = step >> 8;
    MIX_VOL = vol;
}
```

**CPU 真正干的活**：
- 序列调度（哪个通道什么时候变 step）
- ADSR 包络计算（每个音符的 vol 时间曲线）
- 用户输入（按键 → 音高映射）
- 这些用 C 写，glowcc 编译，跑在萤火虫核上

## 六、待解决的设计问题

### A. CPU ↔ 协处理器接口方式
- [ ] 方案 1：内存映射（协处理器寄存器占 RF/RAM 地址段，复用现有总线）
- [ ] 方案 2：专用 IO（加 IO2/IO3，或抢 IO0/IO1 部分位）
- 倾向方案 1（地址译码简单，CPU 用现有 `xx = RAM` 指令访问）

### B. 协处理器状态机实现
- [ ] 方案 1：HC161 + HC138 硬状态机（每状态一组控制信号）
- [ ] 方案 2：小型微码 ROM（32×8 ROM 驱动控制位，类似 CPU 但固定程序）
- 倾向方案 1（5 通道混音的状态固定，不需要可编程）

### C. 乘法 ROM 的查表方式
- [ ] 8×8→8 位无符号：256×256 太大，要 64KB ROM
- [ ] 优化：vol 只 16 级（4 位），可换 (sample×vol) 用 vol 当高 4 位地址 → 256×16 = 4KB ROM（6232 即可）
- [ ] 带符号扩展：乘法 ROM 输出后用 HC86（异或）做补码

### D. 总线仲裁
- CPU 和协处理器都要访问波形 SRAM（CPU 写波形，协处理器读波形）
- [ ] 协处理器独占波形 SRAM，CPU 通过协处理器寄存器间接写
- [ ] 双口 RAM（库存有 7132，但与"直插 + 简单"哲学相悖）
- 倾向前者（CPU 不直接访问 SRAM，所有写入走协处理器寄存器接口）

## 七、与 PSG v0.3 的对照（设计哲学延续）

| 维度 | PSG v0.3 | Glowworm-1 SCC 专用机 |
|------|---------|---------------------|
| CPU | FT232H 主机（外部 PC）| 萤火虫极简核（板载）|
| 音源 | HC161 计数 + TLC7524（纯硬件）| 协处理器（HC161+SRAM+乘法 ROM）|
| 编程 | 主机 PC 写寄存器 | C 程序跑在萤火虫核 |
| 复杂度 | 17 片 | ~30 片 |
| 共通点 | **关键运算交专用硬件，CPU 只调度** | 同 |

Glowworm-1 是 PSG v0.3 思路的延续——只是"CPU"从外部 PC 换成板载萤火虫，"专用硬件"从简单方波计数器扩展到带 SRAM 波形表的混音协处理器。
