# 迷你萤火虫调度核（CPU 最小化设计）

> 整理时间：2026-07-05
> 核心问题：CPU 当调度器，真的需要完整萤火虫（23 片）吗？能不能砍到 10 片左右？

## 一、CPU 当调度器到底要会干什么

把"调度"拆成 ISA 需求（基于 STC ADSR/序列代码分析）：

```c
// 一个采样周期 CPU 的工作（~453 拍预算 @ 8MHz/17640Hz）：
for (ch = 0; ch < N; ch++) {
    MIX_CH     = ch;          // 立即数 → IO
    MIX_STEP_L = step_l;
    MIX_VOL    = vol;
    env_tick(ch);             // RF 读写 + ALU + 跳转（ADSR 算法）
    MIX_START  = 1;
    while (!(MIX_STATUS & 1));// IO → A + 跳转
    mix_acc += MIX_OUT;       // IO → A + 加法
}
DAC = mix_acc >> 2;
```

### ISA 子集（调度核需要的）

| 指令类别 | 用途 | 是否必需 |
|---------|------|---------|
| 立即数 → A/RA | 设通道号、step、vol | ✅ |
| A → IO | 写协处理器寄存器 | ✅ |
| IO → A | 读 MIX_STATUS / MIX_OUT | ✅ |
| RF ↔ A | 包络/累加器变量 | ✅ |
| ALU ADD/SUB/CMP（带进位）| ADSR 计数、phase 累加、比较 | ✅ |
| JMP / JCC | 循环、ADSR 分支 | ✅ |
| **立即数 → A0/A1/A2** | 数据 RAM 寻址 | ❌ 不需要 |
| **A0/A1/A2 → ...** | 同上 | ❌ |
| **B 寄存器** | ALU 第二输入 | ⚠️ 可砍（用 A + 立即数）|
| **long long / double** | 64 位运算 | ❌ |
| **IO1** | 第二 IO 口 | ❌（IO0 够）|
| **RF 分页机制** | 大寄存器堆 | ❌（几个变量够）|
| **完整 25 种 ALU** | MUL/DIV 等 | ❌（乘除交硬件）|

## 二、四个最小化方案对比

| 方案 | 描述 | 兼容 glowcc | 片数 | 可编程性 |
|------|------|-----------|------|---------|
| **A. 完整萤火虫** | 龙少原版 23 片 | ✅ | ~23 | 完整 C |
| **B. 精简调度核** | 砍 A0/A1/A2/B/long long/double，保留核心 ISA | ⚠️ 部分（要改 glowcc 后端）| **~12-15** | C 子集 |
| **C. 自定义迷你 ISA** | 完全新设计，专为调度 | ❌（写汇编/新编译器）| ~8-10 | 汇编 |
| **D. 状态机调度器** | 不用 CPU，HC161+HC138 直接调度 | ❌（无 CPU）| ~5 | 固定逻辑 |

## 三、方案 B 详细设计（精简调度核，~12 片）

### 砍掉的部件（vs 完整萤火虫）

| 部件 | 完整版 | 调度核 | 节省 | 影响 |
|------|--------|--------|------|------|
| A0/A1/A2（24位地址）| 3×HC374 | **砍** | 3 片 | 不能直接访问大 RAM，所有数据走 IO |
| B 寄存器 | 1×HC374 | **砍** | 1 片 | ALU 用 A + 立即数（cmp/add #imm 模式）|
| RF 分页 | 大 SRAM + 寻址 | 小 SRAM 或寄存器堆 | 1-2 片 | RF 只放几个变量 |
| 完整 ALU ROM（16位宽）| 1×IS61LV12816 | 6264 (8K×8) | 同片数但更便宜 | 只存 ADD/SUB/CMP |
| IO1 | 1 套寄存器+译码 | **砍** | 1 片 | IO0 够（8 位 256 个寄存器寻址）|
| long long / double 运算 | ROM 表大量 | **砍** | 0 片（ROM 内容缩减）| 不能跑浮点（不需要）|

### 保留的部件（~12 片）

```
【程序 ROM】   628512 (512K×8)                = 1 片
【数据 RAM】   6264 (8K×8) — RF + 全局变量    = 1 片
【ALU ROM】   6264 (8K×8) — ADD/ADD_C/SUB/SUB_C/EQUAL 几个表
                                                = 1 片 (实际只用 ~320B)
【PC 计数器】  16 位，74161 ×4                  = 4 片
【指令译码】  74138 + 7400 (NAND)              = 2 片
                (原版 4 片 HC138，砍指令后 2 片够)
【寄存器】    A、RA、IR — 74273 ×2 (8D 触发器) = 2 片
【总线驱动】  74244 (三态)                     = 1 片 (可选)

合计：~12 片核心 IC
```

### 与完整萤火虫的对比

| 维度 | 完整萤火虫 | 精简调度核 |
|------|----------|---------|
| 片数 | ~23 | **~12**（省 11 片）|
| ISA | 完整（含 long long/double/24位寻址）| 子集（仅整数 + IO）|
| glowcc 兼容 | ✅ | ⚠️ 需改后端（去掉 A0/A1/A2 等）|
| 可跑 NES/3D/UI | ✅ | ❌ |
| 可跑调度 + ADSR | ✅ | ✅ |
| 主频 | 50MHz（LVC）| 8MHz+（HC 够）|

## 四、方案 D 的诱惑：fsm_player 扩展

fsm_player 是 9 片的纯查表机（无 CPU 概念）：
```
5×HC161 (20位地址计数器) + 2×39SF040 (ROM) + 1×HC273 (锁存) + 1×HC04
```
它能顺序回放 ROM，**但没有 ALU、没有条件跳转**。

如果给 fsm_player 加：
- ALU（1 片 74181 或小 ALU ROM）
- 条件跳转（1 片 HC138 译码 JCC）
- 几个寄存器（1-2 片 HC273）

→ ~12-13 片，**接近方案 B 的功能但起点是 fsm_player 不是萤火虫**。

**问题**：fsm_player 的 ISA 是"地址线性递增"，加跳转后本质就是简化的 CPU——和方案 B 殊途同归。但用 fsm_player 当起点，**没有 glowcc 兼容包袱**，可以设计更紧凑的 ISA。

## 五、关键决策点（你的决策）

### 问题 1：可编程性 vs 片数

- **方案 B（精简萤火虫，~12 片）**：保留 glowcc，ADSR 用 C 写，迭代快
- **方案 D（fsm_player 扩展，~12 片）**：从零设计 ISA，更紧凑但要写汇编或新编译器
- **方案 C（自定义，~8 片）**：极简但要全写汇编

### 问题 2：要不要 glowcc

- glowcc 是龙少现成的，**保留 = ADSR/序列用 C 写**
- 砍掉 = 要么写汇编，要么自己写编译器（工作量大）

### 问题 3：8MHz 主频够吗

调度核每个采样周期 ~453 拍预算，工作内容：
- 5 通道调度（每通道 ~30 拍 IO 写）：150 拍
- ADSR 5 通道（每通道 ~30 拍 C 代码）：150 拍
- 余量 ~150 拍

**8MHz 完全够**，根本不需要 50MHz。

## 六、我的建议

**走方案 B（精简萤火虫，~12 片）**：
1. 复用 glowcc（ADSR/序列用 C 写，开发快）
2. 片数从 23 砍到 12（省 11 片给硬件块）
3. 8MHz 直插够用
4. 改 glowcc 后端的代价可控（删几条指令的代码生成路径）

**下一步具体工作**：
- 列出方案 B 的最终 ISA 子集（保留哪些机器码）
- 估算改 glowcc 后端的工作量
- 设计迷你调度核的接线（用 74161/273/138）
