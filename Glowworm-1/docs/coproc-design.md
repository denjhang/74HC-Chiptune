# 8 大音源运算需求分析（决定哪些硬件化）

> 分析时间：2026-07-05
> 数据源：`STC_Chiptune/STC32G144K246/*.c`（实测出声的代码）
> 用途：决定哪些运算必须硬件化（不能软件跑），就像"显卡"——CPU 调度，运算密集的交专用硬件

## 一、8 个音源按运算特征分类

### 类别 A：方波/计数器型（PSG 同族）—— **运算量小**

**特征**：计数器累加 + period 比较 + 1-bit 输出翻边 + LFSR 噪音

| 音源 | 通道 | 相位 | 输出 | 特殊 |
|------|------|------|------|------|
| **SN76489** | 4 (3方波+1噪音)| 16位计数+period比较 | 1-bit 翻边 × vol | LFSR 噪音（taps 反馈）|
| **AY-3-8910** | 3 方波+噪音 | 16位计数+period比较 | 1-bit 翻边 | 5 位包络发生器 + LFSR |
| **SAA1099** | 6 方波+2噪音 | 16位计数器+八度移位 | 1-bit × vol | 八度移位 + LFSR + 包络 |
| **NES APU** | 2方波+三角+噪音+DPCM | 11位计数+period比较 | duty LUT 查表 | sweep（移位加减）+ duty LUT |
| **Game Boy APU** | 2方波+1波形+1噪音 | 计数+period | 4 位波形查表 | sweep + envelope + 立体声 |

**运算特征**：
- ❌ 无乘法（输出直接 `vol × 符号位`）
- ❌ 无大查表（只有小 duty LUT）
- ✅ 主要是 16 位计数器比较 + 1-bit 翻边
- ✅ LFSR（PSG v0.3 已做硬件版）
- ✅ 可纯软件，但每通道每个采样要做计数比较，**累加到 5-6 通道开销不小**

**硬件化必要性**：低。和 PSG v0.3 同类，PSG 已经验证纯硬件可行（17 片），把这些做成"PSG++"扩展是自然路线。

### 类别 B：Wavetable 型（查表）—— **中等运算量**

**特征**：相位累加 → 高位做索引 → 查表 → 乘音量

| 音源 | 通道 | 相位 | 索引 | 表 | 乘法 |
|------|------|------|------|----|----|
| **SCC** | 5 | 32 位 | `>>16 & 0x1F` | 32 点 | `(b*vol)>>4` |
| **WT (wt.c)** | 4 | **16 位** | `>>5 & 0x7F` | **128 点** | `(b×lvl×vol)>>10` **双乘法** |

**运算特征**：
- ✅ 相位累加（16-32 位）
- ✅ **波形查表（大表，32-128 点）**
- ✅ **乘音量（1-2 次）** ← 实测占 50% 开销

**硬件化必要性**：高（实测验证）。乘法 + 查表是热点，硬件化收益大。

### 类别 C：FM 合成型 —— **运算量极大**

**YM2413**（OPLL）：

```c
// 每采样每通道：
phase_m += step * mul_m * ticks;     // 32 位 + 双乘法
phase_c += step * ticks;
idx_m = (phase_m >> 10) & 0x3F;
idx_m = (idx_m + fb_val) & 0x3F;
mod_out = ym_conv_vol[level_m][idx_m];   // 二维查表（64×64=4KB）
fb_val = mod_out * fb >> 4;                // 乘法（反馈）
idx_c = (phase_c >> 10) & 0x3F;
idx_c = (idx_c + mod_out) & 0x3F;          // 调制器输出加到 carrier 相位
carrier = ym_conv_vol[level_c][idx_c];     // 二维查表
mix += carrier * (16 - vol) >> 2;          // 乘法
```

**运算特征**：
- ✅ **相位累加带乘法**（双乘法 + 32 位）
- ✅ **二维查表**（level × phase 的 4KB 表，是 sine × envelope 的预乘）
- ✅ **反馈调制**（OP1 输出反馈 + 乘 fb）
- ✅ **相位调制**（mod_out 加到 carrier 索引——FM 本质）
- ✅ **每通道两次乘法 + 两次查表**

**硬件化必要性**：极高。FM 的相位调制链（OP1→反馈→OP2→输出）是流水线，软件串行极慢，**硬件并行能 1 拍完成**。

## 二、运算操作分类（决定硬件块设计）

把所有音源的运算抽象成"基本操作"，看哪些值得硬件化：

| 基本操作 | 出现于 | 软件开销 | 硬件实现 | 硬件化收益 |
|---------|--------|---------|---------|----------|
| **16/32 位相位累加** | 全部 | 中（多字节 ALU）| 2-4×74283 | ⭐⭐ 中 |
| **波形查表（一维）** | SCC/WT/NES-duty | 高（地址拼+RAM 读）| SRAM + HC273 | ⭐⭐⭐ 高 |
| **波形查表（二维）** | YM2413 conv_vol | **极高** | SRAM + 地址拼接 | ⭐⭐⭐⭐ 极高 |
| **乘音量（8×8）** | SCC/WT/YM | **极高**（实测大头）| 乘法 ROM | ⭐⭐⭐⭐ 极高 |
| **双乘法 + 移位** | WT/YM（带包络）| **极高** | 2 片乘法 ROM 流水 | ⭐⭐⭐⭐ 极高 |
| **LFSR 噪音** | SN/AY/SAA/NES/GB | 低（移位+异或）| HC164+异或门（PSG 已验证）| ⭐ 可复用 PSG |
| **1-bit 方波翻边** | SN/AY/SAA/NES/GB | 低 | 74161 计数+比较 | ⭐⭐ 中 |
| **DPCM/ADPCM 解码** | NES DPCM/adpcm/brr | 高（差分+移位）| HC283 加法 + 移位 | ⭐⭐⭐ 高 |
| **sweep（移位加减）** | NES/GB | 低 | HC175 移位 + 加 | ⭐ 低 |

## 三、建议的硬件块清单（"音源显卡"设计）

按 ROI（性能收益 ÷ 硬件成本）排序：

### 必做（极高 ROI）—— 共用，所有音源都用得到

**1. 通用波形查表 + 乘音量单元（"wave-mix"单元）**
- 输入：phase（16-24 位）、wave_base、vol、level
- 输出：sample × level × vol（移位后）
- 内部：SRAM（波形表）+ 乘法 ROM × 2（双乘法流水线）
- 服务：SCC、WT、YM2413（OP1/OP2 输出阶段）
- 估计片数：8-10 片（SRAM + 2×乘法 ROM + 74283 累加 + HC374 锁存）
- **收益**：571 拍 → ~30 拍/通道

**2. FM 调制链专用单元（"op-chain"单元）**
- 输入：phase_m、phase_c、level_m、level_c、fb
- 输出：carrier sample
- 内部：2 次二维查表（ym_conv_vol）+ 反馈调制加法
- 服务：YM2413（专用，无法共用）
- 估计片数：5-7 片（2 块 4KB ROM + 加法 + 锁存）
- **收益**：让 FM 在 8MHz 可行（FM 软件根本跑不动）

### 中 ROI（PSG 同族复用）

**3. PSG 计数器组（计数+翻边+period 比较）**
- 输入：period、vol、noise_taps
- 输出：1-bit 信号 × vol（带符号）
- 内部：74161 计数器 + HC85 比较器 + LFSR（HC164+异或）
- 服务：SN76489、AY8910、SAA1099、NES（方波部分）、GB（方波部分）
- 这就是 **PSG v0.3 的扩展版**（PSG 已验证）
- 估计片数：8-12 片/通道组

### 低 ROI（不值得）

**4. sweep 单元**——软件实现够快（移位+加）
**5. DPCM 解码单元**——除非要做采样回放，否则低优先

## 四、整体系统架构（软硬结合的"音源显卡"）

```
┌──────────────────────────────────────────────────┐
│  萤火虫 CPU 核（8MHz，纯调度）                    │
│  - 接收主机命令（MIDI note / 寄存器写）           │
│  - 算 ADSR 包络（C 代码，分支多用 CPU 强项）       │
│  - 调度：选音源类型 → 启动对应硬件块              │
└────┬──────────────┬──────────────┬──────────────┘
     │              │              │
     ▼              ▼              ▼
┌─────────┐  ┌─────────────┐  ┌──────────┐
│ PSG 计数 │  │ wave-mix   │  │ FM op-   │
│ 器组     │  │ 单元       │  │ chain    │
│ (SN/AY/  │  │ (SCC/WT/   │  │ 单元     │
│  SAA/NES │  │  YM 输出)  │  │ (YM2413  │
│  /GB)    │  │            │  │  专用)   │
│ ~10 片   │  │ ~10 片     │  │ ~6 片    │
└────┬─────┘  └─────┬──────┘  └────┬─────┘
     │              │              │
     └──────────────┴──────────────┘
                    │
                    ▼
              ┌──────────┐
              │  混音 +  │
              │  DAC     │
              │ (TLC7524)│
              └──────────┘
```

**总硬件**：CPU 核（~12 片）+ PSG 块（~10）+ wave-mix 块（~10）+ FM 块（~6）+ 混音 DAC（~3）= **~41 片**，落在 30-50 片预算。

## 五、CPU 的角色（"调度器"，不是运算器）

CPU 在每采样的工作（C 代码）：

```c
void tick() {
    // 1. 决定每通道用哪个硬件块
    for (ch = 0; ch < N; ch++) {
        switch (ch_mode[ch]) {
            case MODE_PSG_SQUARE:
                psg_set(ch, period, vol, noise);
                break;
            case MODE_WAVETABLE:
                wave_mix_start(ch, phase, wave_base, vol, level);
                break;
            case MODE_FM:
                fm_op_start(ch, phase_m, phase_c, level_m, level_c, fb);
                break;
        }
    }

    // 2. 算 ADSR（所有音源共用）
    for (ch = 0; ch < N; ch++) {
        env_tick(ch);   // C 代码，~20 拍/通道
    }

    // 3. 收集结果输出
    DAC = mix_all_done();
}
```

CPU 完全不参与运算密集的部分（查表/乘法/相位累加），只做调度和 ADSR。

## 六、实施路线建议

按"硬件块 ROI × 易实现度"排序：

1. **先做 wave-mix 单元**（覆盖 SCC/WT/部分 YM）—— 收益最大，覆盖 3 个音源
2. **PSG 块复用 v0.3 经验**（覆盖 SN/AY/SAA/NES/GB）—— 已有验证，5 个音源
3. **最后做 FM op-chain**（YM2413）—— 最难但独家
4. CPU 端用 glowcc 编译 STC 的 ADSR/序列代码

**这样 8 个音源都能在 8MHz 萤火虫上跑**，且硬件块按"运算密集度"分配——和显卡一个道理。
