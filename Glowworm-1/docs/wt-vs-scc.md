# WT (Wavetable) 路线 vs SCC 路线对比

> 整理时间：2026-07-05
> 来源：STC_Chiptune/STC32G12K128/wt.c（轻量级 wavetable 合成器）
> 用户决策：**不追求寄存器兼容**，可重新设计 API

## 一、WT vs SCC 算法对比

| 维度 | SCC (scc.c) | **WT (wt.c)** |
|------|-------------|--------------|
| 相位累加器 | **32 位** (`u32 scc_cnt`) | **16 位** (`u16 pos`) |
| 波形索引 | `(cnt >> 16) & 0x1F` | `(pos >> 5) & 0x7F` |
| 波形表 | 32 字节/通道 | **128 字节/通道**（14 种预置）|
| 乘法 | `(b * vol) >> 4`（8×8，1 次）| `(wave * (lvl+1) * (vol+1)) >> 10`（**双乘法**）|
| 包络 | 无（外部喂）| **内置 ADSR**（wt_env_tick）|
| 音色 | 外部 | **内置音色模板**（wt_tone）|
| 频率查表 | 实时算 | **预计算表** wt_note_freq[104] |
| 通道数 | 5 | 4 |
| 寄存器协议 | 兼容 K051649 | **全新 API**（用户决策不兼容历史）|

## 二、实测开销对比（glowworm_sim 真实编译跑）

| 算法 | 单通道开销 | 5/4 通道总开销 | 8MHz 预算（453拍）|
|------|----------|--------------|------------------|
| SCC（32位相位+单乘法）| **580 拍** | 5×580 = 2900 拍 | ❌ 差 6.4 倍 |
| **WT（16位相位+双乘法）** | **571 拍** | 4×571 = 2284 拍 | ❌ 差 5.0 倍 |

**结论**：
- WT 略快（16 位相位省 ~25 拍，被多一次乘法吃掉 ~30 拍，净省 9 拍）
- 两者数量级相同，**纯软件都不够，协处理器都必需**
- **乘法是大头**（约 50%），16 vs 32 位相位只省 ~25 拍

## 三、WT 路线对萤火虫的友好点

1. **16 位相位累加器**——萤火虫 `short=2字节`，加法 2 字节 ALU 序列化（不是 4 字节）
2. **预置波形 + 音色模板**——CPU 端用查表（萤火虫查表 ALU 强项），不用实时算波形
3. **预计算频率表**——CPU 收到 MIDI note 直接查表得 step，不用现场除法
4. **内置 ADSR**——CPU 端用 C 写（萤火虫 int=4字节 + 分支预测，CPU 的强项）
5. **不兼容历史寄存器**——可重新设计 API 适配协处理器架构，无包袱

## 四、协处理器加速后的理论开销（WT 路线）

把"波形查表 + 双乘法 + 移位"交给硬件协处理器，CPU 只剩：

| 操作 | CPU 端开销 | 备注 |
|------|----------|------|
| `pos += step`（16 位加法）| ~50 拍 | 2 字节 ALU 序列化 |
| 启动协处理器 | ~5 拍 | 写 MIX_START 寄存器 |
| 等待 + 读结果 | ~10 拍 | 协处理器跑 ~30 拍（硬件并行查表+乘法）|
| **单通道总开销** | **~65 拍** | vs 571 拍纯软件，**省 89%** |

**4 通道 WT 8MHz 实时性**：
- 总开销 = 4 × 65 = **260 拍/采样**
- 8MHz 预算 = 453 拍/采样
- **余量 43%** ✅

CPU 还能跑 ADSR + 序列调度（每采样 ~190 拍预算）。

## 五、协处理器硬件实现（用 161/283/181）

WT 协处理器核心 = 16 位相位累加 + 128 字节波形查表 + 双乘法：

```
【16 位相位累加器】
  - 2×74283（4 位加法器级联）= 16 位组合加法，1 拍出全和
  - 2×74161（4 位计数器当锁存）= 16 位相位寄存器
  - 或：2×74283 + 2×HC374（4 片/16 位）

【128 字节波形表】
  - 6264 SRAM（8K×8）或 62256
  - 地址 = {ch[1:0], idx[6:0]}（4 通道 × 128 = 512B）

【双乘法 + 移位】
  方案 A：乘法 ROM（推荐）
    - (wave × level × vol) >> 10
    - level 5 位、vol 5 位、wave 8 位 → 18 位地址，太大
    - 优化：分两步查表
      step1: wave × level → 8 位 ROM（256×8 = 2KB，1 片 6116）
      step2: result × vol → 8 位 ROM（256×8 = 2KB，1 片 6116）
    - 共 2 片 ROM + 移位（HC175 选高 8 位）

  方案 B：74181 ALU 移位累加（8 轮）
    - 慢但可编程，复用性高
    - 2×74181（处理 4 位段）

【累加器】
  - 4 通道混音：4×127 = 508 < 1024，10 位够
  - 3×74161（计数器模式累加）或 2×74283 + 锁存

【通道计数器 + 状态机】
  - 1×74161（2 位通道计数 0-3）
  - 1×HC138（状态译码）
```

**协处理器片数估算**：12-15 片（与 SCC 协处理器同级）

## 六、WT 路线的 API 设计（不兼容历史）

由于不追求 SCC 寄存器兼容，可重新设计 API 适配协处理器：

```c
// 协处理器寄存器（CPU 视角，内存映射）
extern u8 MIX_START;        // 写 1 启动一轮 4 通道混音
extern u8 MIX_STATUS;       // bit0=DONE
extern u8 MIX_OUT;          // 当前通道的混音结果

// 通道参数（CPU 写入，协处理器读取）
extern u8 WT_CH;            // 通道号（设参数时用）
extern u8 WT_STEP_L, WT_STEP_H;   // 16 位步进
extern u8 WT_LEVEL;         // ADSR 输出（CPU 算）
extern u8 WT_VOL;           // 主音量
extern u8 WT_WAVE_IDX;      // 波形号（0-13）

// 协处理器一个采样周期：
void wt_tick() {
    for (u8 ch = 0; ch < 4; ch++) {
        wt_env_tick(ch);            // CPU 端算 ADSR（C 代码）
        WT_CH = ch;                 // 选通道
        WT_LEVEL = wt_ch[ch].level; // 写入 ADSR 结果
        MIX_START = 1;              // 启动协处理器
        while (!(MIX_STATUS & 1));  // 等 DONE
        DAC = MIX_OUT + 128;        // 输出
    }
}
```

## 七、决策

**WT 路线优于 SCC 路线**（对萤火虫）：
1. 16 位相位省硬件（少 2 片 74283）
2. 不兼容历史 = API 干净
3. 内置 ADSR/音色模板更完整
4. 实测开销两者相当

**建议**：把 Glowworm-1 的目标从"SCC 专用机"调整为 **"WT (Wavetable) 专用机"**，参考 wt.c 算法 + 不兼容 API。
