# SCC 在萤火虫架构上的性能基准（实测）

> 测量时间：2026-07-05
> 方法：用 glowcc 编译 STC 移植的 SCC 算法 → glowworm_sim 跑真实 ROM → 数周期
> 程序：`sw/scc1ch/main_unrolled.c`（单通道混音循环手动展开 5 次，防 lcc 死代码消除）

## 实测结果

### 单通道 SCC 混音开销

```
每次 DO_MIX() = 580 拍
```

`DO_MIX` 内容（移植自 STC scc.c 的 SCC_MIX_CH）：
```c
cnt += step_val;                          // 32 位加法（萤火虫 int=4字节，4 次字节 ALU）
offs = (cnt >> 16) & 0x1F;                // 32 位右移 + 与
b = wav[offs];                            // 数组查表
if (b >= 128)                             // 比较 + 分支
    tmp = -((256-b) * vol) >> 4;          // 8×8 乘法 + 移位 + 取反
else
    tmp = (b * vol) >> 4;
mix += tmp;                               // 16 位累加
```

### 性能推算（萤火虫纯软件 SCC）

| 主频 | 单通道采样率上限 | **5 通道采样率上限** | vs SCC 标准 (17640 Hz) |
|------|----------------|---------------------|----------------------|
| 8 MHz | 13793 Hz | **2758 Hz** | ❌ 差 6.4 倍 |
| 50 MHz | 86206 Hz | **17241 Hz** | ⚠️ 刚好够（无余量）|

**结论验证**：8 MHz 纯软件跑 5 通道 SCC 远不够（你早就判断了），**必须特化指令集或硬件**。

## 580 拍开销的来源分析

萤火虫 `int = 32 位`，所有 32 位运算靠 4 字节 ALU 查表 + 进位链序列化：

| 操作 | 估计拍数 | 备注 |
|------|---------|------|
| `cnt += step_val`（32位加法）| ~50 | 4 字节 ADD + 3 字节 ADD_C 进位链 + RAM 读写 |
| `cnt >> 16`（32位右移）| ~80 | 编译器展开成字节搬运 |
| `& 0x1F` | ~10 | 1 字节 AND |
| `wav[offs]`（数组寻址）| ~60 | 基址 + 索引 + RAM 读 |
| `b >= 128` 比较 + 分支 | ~30 | |
| `(256-b) * vol` 或 `b * vol` | ~150 | 8×8 乘法（萤火虫有 MUL_L 单指令但 lcc 可能展开）|
| `>> 4` + 取反 + 16位累加 | ~100 | |
| 临时变量 RF 读写 | ~100 | lcc 寄存器分配 spill |
| **合计** | **~580** | 与实测吻合 |

## 特化方向（基于实测开销）

### 三条加速路线

实测 580 拍/采样里，**32 位加法（~50 拍）+ 8×8 乘法（~150 拍）是大头**——这俩正是硬件 ALU（74283 加法器 / 74181 4 位 ALU）天生擅长的。这给出三条加速路线：

| 路线 | 做什么 | 芯片代价 | 5 通道 8MHz 够吗 | 灵活性 |
|------|-------|---------|---------------|-------|
| **A. 纯软件** | 现状不动 | +0 | ❌（2758 Hz，差 6.4 倍）| 最高 |
| **B. 专用指令集** | 加相位累加/查表/乘音量复合指令（微码层面）| +0（重编 ROM）| 待测 | 高（仍可编程）|
| **C. 借助硬件 ALU** | 加 74283/74181 做并行加法/乘法，CPU 当调度器 | +2-4 片 | 待测 | 中（硬件加速块固定）|
| **D. B+C 组合** | 硬件 ALU 加速 + 专用指令调用它 | +2-4 片 + ROM 重编 | 最有希望 | 中 |

**纯软件（A）已被实测排除**。单靠专用指令（B）能省但幅度有限（仍是 8 位序列化）。**硬件 ALU 加速（C/D）很可能是必需的**——这与 PSG v0.3 的设计哲学一致：关键运算交给专用硬件（PSG 用 TLC7524 当 DAC、HC161 当计数器），CPU 只做调度。

### 硬件 ALU 加速的潜力（理论估算）

| 操作 | 纯软件（现状）| +74283 加法器 | +74181/乘法 ROM |
|------|-------------|--------------|----------------|
| 32 位加法（4 字节序列化 + 进位链）| ~50 拍 | **~5 拍**（4×74283 并行，1 拍出全 32 位和）| 同左 |
| 8×8 乘法（移位加法 8 轮）| ~150 拍 | ~80 拍（用 74283 累加）| **~10 拍**（ROM 查表 256×256→16 位）|
| 单通道混音预估 | 580 拍 | ~340 拍 | **~250 拍** |

5 通道 8MHz 下：
- A（纯软件）：2758 Hz ❌
- C（+74283）：~4700 Hz ❌（仍不够）
- D（+74283 + 乘法 ROM）：~6400 Hz ❌（接近但仍不够）
- 50MHz 下 D：~40000 Hz ✅（充裕）

**推论**：8MHz 要跑 5 通道 SCC，可能要更激进的特化（如专用 SCC 混音协处理器，CPU 只喂数据）；或者把 SCC 渲染降采样率（不是每拍都 5 通道全开）。50MHz + 适度硬件加速（路线 D）够用。

### 高 ROI 特化（路线 B/C/D 都适用）

1. **专用"相位累加"指令**（替代 32 位序列化加法）
   - 路线 B：用微码封装 4 字节加法循环
   - 路线 C/D：硬件 74283 并行加法，CPU 单条指令发起

2. **专用"波形查表"指令**（替代数组寻址的多步序列）
   - 节省基址+索引+RAM 读的多步序列

3. **8×8 乘法确保用 MUL_L 单指令**
   - 萤火虫已有 MUL_L/MUL_H ALU 模式（查 ROM），但要确认 lcc 生成的代码真用了它
   - 若 lcc 把乘法展开成移位加法，可能要加专用乘法指令或硬件乘法器

### 中 ROI 特化

4. **16 位数据类型优化**（step/cnt 用 16 位而非 32 位）
   - 萤火虫 short=2 字节，运算序列化次数减半
   - SCC 标准用 16 位相位累加也够（精度略损）

5. **位域提取专用指令**（`(cnt>>16) & 0x1F`）
   - 硬件直接取 32 位寄存器的高 16 位低 5 位

### 低 ROI（不推荐）

6. 5 通道硬件并行（专用 5 通道混音器）—— 失去通用性，违背"特化指令集但保持可编程"原则

## 结论

**纯软件跑 SCC 很难**——580 拍/采样里乘法和 32 位运算占大头，8 位序列化是根本瓶颈。可能的出路：

1. **改指令集**（路线 B）：加专用相位累加/波形查表/乘法指令，仍可编程
2. **借助硬件 ALU 加速**（路线 C/D）：加 74283 并行加法器 / 74181 ALU / 乘法 ROM，CPU 当调度器
3. **B+D 组合最有希望**：硬件加速 + 专用指令调用硬件

这呼应了 PSG v0.3 的设计哲学——关键运算交专用硬件，CPU 只做调度。
